## 1. 这条切片究竟要证明什么

E1 不是做一个普通 CLI demo，也不是横向铺开五系统。

后续 E3–E5 要实现的第一条受控切片，必须能在一个隔离的测试项目中完整表达以下因果链：

```
用户的有界目标
→ 本地登记为正式任务
→ 读取 Git/文件的项目事实
→ 外置后端提交"候选"，而非直接拥有事实
→ 本地 Gate 审查一次低风险行动
→ 执行一个允许范围内的真实文件改动
→ 产生动作收据并重新观察项目
→ 运行预先声明的验证
→ 本地写出带缺口的完成裁决和证据链
```

本切片只能证明"受控行动与真实性链的最小骨架"。它不证明恢复、第二轮经验、完整模型后端、复杂授权、并发项目或初循环 IL-G0–IL-G9 已全部成立。

## 2. E1 已作出的设计决定

| 决策 | E1 冻结结论 | 原因与边界 |
|------|------------|-----------|
| 主语言 | Python 3.12+；最低兼容版本写入 `requires-python = ">=3.12"` | 旧 FurinaOS 工程已有 Python 实践；Python 标准库可直接提供 sqlite3、subprocess、pathlib、hashlib 与 tomllib |
| 代码形态 | `src/furina_code/` 的可安装 Python 包 + `python -m furina_code` CLI | 让运行入口、测试和后续后端适配共享同一应用边界 |
| 正式事实存储 | SQLite 单文件账本，位于受控 runtime 数据目录；只由本地拥有器写入 | SQLite 可提供事务性本地存储；数据库是承载，不是权威。E1 不允许 ORM 或外部数据库 |
| 项目观察 | Git 命令与文件哈希；通过参数列表调用子进程，shell=False | Git HEAD/status/diff 是项目现实材料，不能被后端自述替代 |
| 外置认知入口 | 文件化候选桥接：外置本地 AI/后端生成受版本约束的候选 JSON；本地系统只接受、验证、记录或拒绝 | 不在第一切片绑定某一模型供应商、Agent 框架或密钥。后端仅为 PROPOSER，不拥有任务、行动或完成权 |
| 外置行动入口 | 本地受控执行器按已批准的 ActionProposal 执行；第一版只允许测试工作区中的单文件 create | 行动必须由本地 Gate 约束，并留下动作收据；禁止任意 shell、补丁应用、删除、网络和 Git 写入 |
| 运行时依赖 | 零第三方运行时依赖 | 初始目标是权威链，而不是框架集成 |
| 开发验证依赖 | pytest，版本范围由 E2 固定；不使用 pytest 插件 | 测试可读、可扩展；仅用于开发验证 |

## 3. 第一条真实任务的固定范围

### 3.1 任务样本

在由测试创建的临时 Git 项目中，处理一项明确用户目标：

> "在允许目录 `notes/` 中创建 `welcome.txt`，内容必须完全等于指定文本；不得修改其他项目文件。"

这是一个真实文件系统与 Git 工作区改动，不是内存模拟；但它被隔离在测试临时目录中，且单文件、可观察、可逆、可验证，适合作为首次行动边界。

### 3.2 固定成功条件

只有以下事实同时成立，系统才可写出 `completed_with_evidence`：

1. 原始目标、允许作用域和成功文本被记录为 TaskDossier；
2. 改动前和改动后的 Git/文件观察均已记录，且项目引用一致；
3. 候选行动恰为 create notes/welcome.txt，内容哈希与任务目标一致；
4. 本地 Gate 显式允许该动作，票据未过期、未撤销且作用域匹配；
5. 动作收据记录了实际路径、前后哈希、执行结果及关联 ID；
6. 重新观察证明目标文件存在、内容精确匹配、允许范围外没有文件变化；
7. 预先声明的 pytest 验证实际通过；
8. 完成裁决引用任务条件、观察、收据和验证证据，并明确本轮不覆盖恢复与经验。

任何一项不成立，必须产生 not_completed、partially_supported 或 integrity_uncertain 之一。

## 4. 最小对象与模块切片

| 模块/责任 | 必须最小实现的对象或行为 | 明确不做 |
|-----------|------------------------|----------|
| identity | RunBinding 与相关联 ID 的最小创建/读取 | 多用户、跨项目、身份管理 UI |
| task | TaskDossier、单一 TaskRun 的创建与受限阶段转换 | 完整任务规划、任务队列、并发 |
| ledger | 追加 EventEnvelope、不可原地修改、expected_revision 冲突拒绝 | 通用事件总线、分布式事件存储 |
| world | 目标工作区的 Git HEAD/status 与允许文件哈希快照 | 任意命令、网络、非 Git 项目适配 |
| candidate | 从候选 JSON 接受或拒绝 CandidateEnvelope，记录后端来源标识 | 直连模型 API、自动上下文外发 |
| gate | 静态低风险策略：单项目、单文件创建、notes/、确切文件名/内容、无既存文件覆盖 | 通用策略语言、高风险授权、用户界面 |
| action | 已批准动作的受控单文件创建与 ActionReceipt | 覆盖、删除、重命名、Git 提交、shell、批量改动 |
| verification | 内容精确匹配 + 范围检查 + pytest 结果登记 + CompletionVerdict | 用户主观验收、部署、CI、全量质量体系 |
| report | 机器可读结果摘要，列出已证明、未证明、失败/未知项 | 自然语言聊天界面、经验写入 |
