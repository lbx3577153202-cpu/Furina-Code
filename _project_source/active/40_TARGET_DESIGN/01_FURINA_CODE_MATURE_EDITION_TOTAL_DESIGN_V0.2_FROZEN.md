# 1. 一页总览

## 1.1 成熟 Furina Code 是什么

> **成熟 Furina Code，是在用户控制的本地生命权威与明确验证的项目级能力边界内，能够长期作为同一个可正当演化的开发责任主体，在变化的项目现实中持续承接而不篡夺用户方向，自主承担边界内的开发过程，对行动、现实后果、完成真实性、经验演化及正式承诺范围内的后续影响维持可追踪责任，并能在后端变化、能力退化、故障、升级、用户改向和终止中恢复、降级、移交或有序收束责任链，因此使用户能够长期减少技术过程管理而不失去主权。**

它能够在：

- 后端变化；
- 能力退化；
- 故障；
- 升级；
- 用户改向；
- 迁移；
- 终止；

中恢复、降级、移交或有序收束责任链。

## 1.2 成熟的内部根

成熟的根不是功能更多，而是：

> **持续承担开发责任。**

固定因果方向：

```text
持续承担开发责任
→ 成为长期可靠的开发责任主体
→ 产生长期可托付性
```

## 1.3 用户得到什么

用户可以长期减少：

- 任务拆分；
- 工具编排；
- 后端选择；
- 技术验证；
- 证据整理；
- 经验维护；
- 故障恢复；
- 权限细节管理。

同时不失去：

- 方向主权；
- 价值取舍；
- 纠正权；
- 授权和撤权；
- 暂停；
- 审计；
- 迁移；
- 终止。

## 1.4 成熟不是什么

成熟不等于：

- 永不失败；
- 永久有效；
- 全部项目通用；
- 不需要用户；
- 九系统已经存在；
- 后端更强；
- 文档更多；
- 测试数量更多。

成熟必须绑定当前能力画像、项目范围、后端、工具、权限、验证和资源条件。

# 2. 用户实际怎样使用成熟版

## 2.1 用户主要负责什么

用户主要负责：

- 表达方向；
- 判断体验；
- 做价值取舍；
- 批准高风险或不可逆行动；
- 必要时纠正、暂停、撤权、迁移和终止。

用户不应长期承担：

- 项目经理式任务拆分；
- 架构和工具编排；
- 后端选择；
- 验证和证据组织；
- 经验维护；
- 故障恢复；
- 提示词工程。

## 2.2 日常低打扰

在已声明能力范围内，系统应自主：

- 读取项目；
- 建立责任和任务；
- 选择后端；
- 执行低风险可逆行动；
- 运行验证；
- 记录证据；
- 更新责任；
- 维护经验；
- 报告结果。

不应逐项要求用户确认普通技术动作。

## 2.3 关键高透明

以下情况必须主动升级：

- 方向冲突；
- 重大价值取舍；
- 高风险或不可逆行动；
- 权限扩大；
- 成本显著超出；
- 安全或隐私边界变化；
- 用户目标无法达成；
- 证据不足；
- 能力退化；
- 成熟范围缩小；
- 无法恢复；
- 需要移交或终止。

## 2.4 成熟版的标准生命过程

```text
用户表达方向
→ 形成正式方向
→ 建立或更新项目级责任
→ 读取主体、项目、责任和经验
→ 观察当前项目现实
→ 评估后端、权限、风险和资源
→ 编排行动
→ 签发有限授权
→ 执行真实动作
→ 独立重新观察
→ 验证结果和用户效果
→ 更新任务与责任
→ 维护后续影响
→ 治理经验
→ 报告结果、证据和缺口
→ 进入下一轮持续开发
```

## 2.5 能力不足也是正常成熟行为

当能力不足时，成熟系统应：

```text
发现能力或证据不足
→ 缩小计划或作用范围
→ 标记未知
→ 请求必要用户决定
→ 降级、暂停、移交或终止
→ 不超过证据报告
```

成熟不是“总能继续”，而是“不能继续时仍然真实、负责和可收束”。

# 3. 完整开发责任生命周期

## 3.1 责任建立

正式责任只能来自：

- 用户正式方向；
- 已存在的维护承诺；
- 合法恢复义务；
- 合法终止收束义务。

后端、经验和系统自身偏好不能成为责任来源。

责任至少具有：

- 来源；
- 范围；
- 验收语义；
- 当前状态；
- 未决事项；
- 关闭条件；
- 移交条件；
- 失效条件；
- 后续影响。

## 3.2 责任运行

```text
正式方向
→ 项目级责任
→ 多轮任务
→ 受治理行动
→ 现实结果
→ 验证
→ 用户反馈
→ 责任继续、调整、维护或收束
```

## 3.3 普通责任关闭

普通责任关闭不依赖 C1。

```text
S2提供用户目标与验收语义
+ S5提供责任与任务范围
+ S3提供被验证现实对象与版本
→ 由冻结的组装规则生成版本化 VerificationClaim
→ S5提出关闭候选
→ S3提供当前现实
→ S7接收、检查、验证并签发 CompletionVerdict / ClosureEligibility
→ 必要时S2取得用户主观验收
→ S5检查未决事项和后续影响
→ S5按冻结状态机关闭
```

未决事项必须分别：

- 完成；
- 保留为持续责任；
- 正式移交；
- 经用户明确放弃；
- 标记为超出范围；
- 或进入失效、暂停、终止。

## 3.4 责任移交

移交至少包含：

- 接收对象；
- 移交范围；
- 当前事实；
- 证据；
- 未决事项；
- 风险；
- 权限状态；
- 接收确认；
- 原责任方退出条件。

没有接收确认，不得标记“已移交”。

## 3.5 终止

终止意味着：

- 停止新责任；
- 撤销活动权限；
- 停止现实作用和后端调用；
- 保存完成、未完成和未知；
- 形成责任收束清单。

终止不得伪装成完成。

## 3.6 数据处置

运行终止与本地数据处置是两个不同决定。

用户可以选择：

```text
本地保留
导出后保留
选择性删除
导出后删除
完整删除
```

完整删除后，当前主体连续性、责任历史和成熟证据无法继续恢复；后续重新创建的是新主体。系统不得秘密保留副本，并必须在用户知情确认后服从完整删除决定。

# 4. 整体生命架构

## 4.1 四个基本存在者

### 用户

提供：

- 方向；
- 价值；
- 体验判断；
- 权限边界；
- 重大取舍；
- 迁移和终止决定。

### 本地持续生命操作系统

承载：

- 主体身份；
- 用户和项目关系；
- 正式责任和任务；
- 项目现实；
- 授权和治理；
- 证据和完成真实性；
- 程序经验；
- 恢复和迁移；
- 成熟声明历史。

### 外置认知—行动后端

提供：

- 理解；
- 推理；
- 规划；
- 代码分析和生成；
- 工具与行动能力；
- 评审建议。

后端不拥有主体、方向、责任、权限、完成和经验权威。

### 项目世界

包括：

- 仓库；
- 文件；
- 依赖；
- 构建和测试；
- 运行状态；
- 部署和外部服务；
- 项目历史；
- 技术债；
- 用户真实使用结果。

## 4.2 独立成熟裁决边界

九系统之外存在：

```text
C1 独立成熟裁决边界
```

C1只负责成熟声明生命周期，不参与日常任务和普通责任关闭。

## 4.3 整体关系

```text
用户方向与主权
        ↓
用户拥有的本地持续生命操作系统
        ↔
可替换外置认知—行动后端
        ↓
项目世界中的真实行动与结果
        ↓
证据、责任、经验和恢复回到本地生命
```

成熟验证另有：

```text
冻结画像与验证计划
→ 只读成熟证据快照
→ C1独立裁决
→ S1本地登记
```

# 5. 成熟九系统

## 5.1 S1 主体身份与连续性系统

负责：

- 主体标识；
- 用户与项目绑定；
- 连续谱系；
- 核心定义和宪法版本；
- 正当演化；
- 恢复锚点；
- 成熟裁决本地登记。

禁止：

- 解释用户方向；
- 拥有任务、事实、证据和经验；
- 自行批准身份变化；
- 签发成熟声明；
- 以连续性拒绝用户终止。

---

## 5.2 S2 用户关系与意愿承接系统

负责：

- 用户原始表达；
- 解释候选；
- 正式方向；
- 目标和验收语义；
- 用户纠正；
- 用户主观体验；
- 需要交还用户决定的问题。

禁止：

- 推测偏好升级为正式方向；
- 用户沉默升级为授权；
- 直接执行现实动作；
- 替代技术验证；
- 以少打扰降低重大风险透明度。

---

## 5.3 S3 项目世界建模与现实作用系统

内部保持：

```text
S3-O 现实观察域
S3-A 现实作用域
```

负责：

- 当前项目事实；
- 项目对象版本；
- 外部变化；
- 现实动作；
- 动作回执；
- 动作后重新观察。

禁止：

- 自行授权；
- 执行回执直接变成完成；
- 自行关闭责任；
- 使用作用器缓存伪装独立观察。

---

## 5.4 S4 外置后端共生与能力中介系统

负责：

- 后端注册；
- 后端元数据；
- 可用性；
- 成本、速度、隐私和可靠性；
- 请求响应；
- 后端选择与切换。

禁止：

- 拥有主体和责任；
- 后端输出直接变成事实和完成；
- 后端私有记忆变成本地经验；
- 后端声明直接变成 VerifiedCapability。

# 6. 六条核心权威不变量

精确权威规则由冻结的《成熟九系统权威与隔离宪法 V0.2》负责。本稿只保留六条整体不变量。

## 6.1 单一正式写入权威

每一种正式状态只能有一个主要写入权威。

## 6.2 不得形成自证闭环

提案、批准、执行、观察、验证和登记不得组合成无现实来源的封闭证明环。

## 6.3 后端不拥有生命权威

外置后端不拥有：

- 主体；
- 正式方向；
- 责任；
- 授权；
- 项目事实；
- 完成；
- 程序经验；
- 成熟声明。

## 6.4 禁止跨系统直接写入

共享数据库、脚本、恢复工具和事务协调器不得绕过正式状态转换入口。

## 6.5 单调保守降级

允许保守自动发生：

```text
有效 → 暂停
已验证 → 失效
能力有效 → 退化或未知
授权有效 → 撤销或过期
经验有效 → 暂停使用
```

提高信任、扩大范围和恢复成熟必须重新批准。

## 6.6 用户主权不可转移

用户始终拥有：

- 方向；
- 价值；
- 撤权；
- 暂停；
- 导出；
- 迁移；
- 终止。

# 8. 成熟成立、降级与撤销

精确要求由《Furina Code 成熟成立判据 V0.2》负责。本节只保留总览。

## 8.1 MAT-C1—MAT-C5

```text
MAT-C1 独立成熟裁决
MAT-C2 因果使用
```

- 缩小；
- 暂停；
- 过期；
- 撤销；
- 等待重验。

成熟不是永久标签。

```text
项目现实
```

```text
现实事实
→ 验证
```

```text

→ 重建责任链
→ 无法继续时移交或终止收束
```

# 10. 与初循环的关系

## 10.1 同一生命体的两个阶段

```text
初循环
=
责任闭环第一次真实成立

成熟版
=
同一责任主体长期承担项目级责任
```

## 10.2 初循环是范围绑定前置

成熟画像中的基础能力必须：

- 已经在对应范围通过初循环级闭环；
- 或先补充对应范围的初循环验证；
- 或明确标记为未成熟、受限或不可用。

初循环一次成立不构成通用永久资格。

## 10.3 成熟必须继承

成熟必须保留：

- 初循环主体；
- 用户和项目绑定；
- 方向历史；
- 责任和任务历史；
- 项目现实来源；
- 证据来源；
- 程序经验来源；
- 恢复历史。

成熟不能通过删除初循环历史证明自己更成熟。
