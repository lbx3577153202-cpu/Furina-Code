## 0. 冻结声明

本文件冻结的是：

1. Furina Code 初循环的五系统骨架；
2. 初循环成立所必须闭合的五条生命因果链；
3. G0—G9 十道硬 Gate；
4. T1—T7 七类证明义务；
5. L0—L4 成立等级；
6. 初循环成立声明的范围边界；
7. 防止伪连续、伪行动、伪证据和伪成长的最低约束。

本文件不冻结：

- 五系统内部器官；
- 具体状态机实现；
- 类名、文件名、目录名和数据库结构；
- 外置后端供应商或 API；
- 测试框架；
- 存储技术；
- 成熟九系统的完整工程形态。

文中对象名、状态名和流程名表达的是**必须能够区分的语义**，不要求工程实现采用相同的字段名、枚举名或代码结构。

冻结不代表已经实现。只有本文件规定的证据全部形成并通过裁决，才允许在对应能力范围内声称初循环成立。

## 1. 文档定位

本文件只回答一个问题：

> 在受限但真实的本地项目环境中，Furina Code 至少必须完成什么样的生命闭环，才可以被判定为“初循环成立”？

初循环不是：

- 一次模型调用；
- 一次代码修改；
- 一次测试通过；
- 一份完整架构文档；
- 五个目录或五个类；
- 一次 dry-run 或预览；
- 一个能够保存 JSON 状态的自动化代理。

初循环成立意味着：

> Furina Code 能够从用户拥有的本地权威状态恢复同一主体、用户方向和任务生命，通过可替换的外置认知—行动后端推进真实开发任务，使后端的认知与行动始终进入本地主权、现实、验证和经验闭环，并让第一轮经过验证的经历真实改变第二轮行为。

---

## 2. “同一个本地主体”的操作性定义

“同一个 Furina Code”不是指：

- 新模型模仿旧模型的语气；
- 重新加载一段角色提示词；
- 根据聊天摘要继续扮演；
- 使用同一个后端账号或会话。

同一个本地主体至少由以下连续谱系共同确定：

```text
稳定的主体标识
+
同一用户与项目绑定
+
连续的用户意愿版本谱系
+
连续的任务状态谱系
+
连续的现实事实、授权、裁决与经验引用关系
```

模型、供应商、会话和提示词实例可以更换；上述本地谱系不能因此断裂或被云端会话替代。

---

## 3. 初循环五系统

### I1　主体连续与恢复系统

负责维持并恢复：

- Furina Code 的稳定本地主体标识；
- 用户与项目绑定；
- 当前有效用户方向及其版本引用；
- 各系统权威状态的持久化版本；
- 任务现场和恢复锚点；
- 启动、关闭与崩溃后的基础恢复；
- 主体、任务、事实、裁决与经验之间的连续引用关系。

I1 负责持续性和恢复，不拥有其他系统业务状态的最终解释权。

例如：

- I2 决定任务当前处于哪个业务阶段；
- I3拥有现实观察；
- I4拥有行动授权与完成裁决；
- I5拥有程序经验的晋升状态；
- I1只保证这些权威状态可以一致地保存、定位和恢复。

---

### I2　任务运行与后端调用系统

负责：

- 建立和维护本地任务；
- 保存任务目标、验收条件引用、阶段、执行位置和未完成事项；
- 编译外置后端所需的工作上下文；
- 调用外置认知—行动后端；
- 接收后端分析、计划、工具调用请求和结果描述；
- 推进、暂停、重试、恢复和关闭任务；
- 在后端会话销毁或替换后接续同一任务。

任务属于本地系统，不属于后端会话。

外置后端提供认知与行动能力，但不能直接拥有或改写：

- 用户正式目标；
- 本地任务生命周期；
- 行动授权；
- 项目现实事实；
- 完成裁决；
- 正式程序经验。

---

### I3　项目现实感知与作用系统

负责：

- 观察本地仓库、文件、Git、环境、进程、测试和运行状态；
- 向经过授权的外置行动后端或本地执行器提供受控现实能力；
- 执行或承载经过授权的文件修改、命令、测试、构建、checkpoint 和回滚；
- 保存原始动作回执；
- 行动后重新独立观察现实；
- 标记事实的来源、时间、项目状态指纹和新鲜度。

外置后端可以通过 I3 暴露的受控工具通道真实行动，但不得绕过 I4 授权和 I3 现实边界，直接获得无约束的本地写入能力。

允许两种行动形式：

```text
形式 A
后端提出结构化行动请求
→ I4 授权
→ I3 执行

形式 B
I4 授予限定能力
→ 外置行动后端通过 I3 工具通道执行
→ I3 记录动作并重新观察现实
```

无论采用哪种形式，I3只回答“现实发生了什么”，不能自行宣布用户目标完成。

---

### I4　行动授权与完成裁决系统

I4 在初循环阶段是一个一级系统，但内部必须保持两条逻辑独立、不可相互替代的权威链。

#### I4-A　行动授权域

判断：

- 动作是否在用户授权范围内；
- 是否越界；
- 是否具有破坏性；
- 是否可逆；
- 是否需要 checkpoint；
- 是否需要用户确认；
- 是否应拒绝、限制、暂停或降级；
- 可以向外置行动后端授予哪些限定工具能力。

输出语义为 `ActionAuthorization`。

#### I4-B　完成裁决域

判断：

- 哪些事实已经被 I3 独立观察；
- 证据支持到什么范围；
- 哪些测试实际运行过；
- 是文件已修改、实现存在、行为已验证，还是用户目标已达成；
- 哪些内容仍未证明；
- 正式结果最多可以声称到什么程度。

输出语义为 `CompletionVerdict`。

I4-A 与 I4-B 必须满足：

- 使用不同的输入对象；
- 保存不同的裁决记录；
- 行动授权不能被当作完成证据；
- I4-B 只能依据正式验收条件和 I3 现实事实裁决；
- I4-A 不能预先保证行动结果正确；
- 同一个未经隔离的判断过程不得既批准行动，又依据自身输出宣布成功。

---

### I5　经验巩固与程序性成长系统

负责：

- 关联任务轨迹、行动、失败、成功和现实结果；
- 提炼候选程序经验；
- 保存适用条件、依赖前提、禁止条件、风险、证据引用和失效信号；
- 在后续相似任务中检索并实际使用经验；
- 记录经验具体改变了哪个计划、检查顺序或行动；
- 根据复用结果对经验进行保留、修正、降级或废弃。

保存聊天、日志或摘要不等于成长。

只有以下链条成立，才形成程序性成长：

```text
经过验证的经历
→ 候选经验
→ 适用性检查
→ 后续任务实际采用
→ 行为产生可观察变化
→ 再次验证
→ 经验状态更新
```

## 4. 初循环允许的能力范围

V0.2 允许初循环在明确受限范围内成立，例如：

- 单一用户；
- 单一本地项目；
- 任一时刻只有一个活动主任务；
- 完整验证期至少包含两个相关但不相同的任务实例；
- 简单线性子任务；
- 一个主要外置后端；
- 可完全销毁并重新建立后端会话；
- 低风险、可逆的文本或代码修改；
- 受控命令集合；
- 基础 checkpoint；
- 进程重启后的恢复；
- 一种简单的程序经验提炼与检索机制。

V0.2 的 L3 硬门槛要求：

- 后端接入具有明确接口边界；
- 任务状态不属于后端会话；
- 旧会话可以完全销毁；
- 干净的新会话可以只依靠本地正式状态接续任务。

以下属于更强证据或 L4 稳定化目标，不是 L3 硬门槛：

- 在不同供应商或不同模型族之间实际切换；
- 多用户权限体系；
- 多项目并行；
- 多后端复杂智能路由；
- 跨设备完整迁移；
- 高风险自主操作；
- 长周期无人值守运行；
- 多进程分布式一致性；
- 大规模经验冲突治理；
- 完整恶意提示注入防御；
- 成熟九系统全部分化。

任何成立声明必须绑定实际验证范围，不能从一个受限任务外推为通用自主开发能力。

## 5. 五条必须闭合的生命因果链

### 5.1 意愿链

```text
用户原始表达
→ 系统解释
→ 正式目标
→ 验收条件
→ 本地任务
→ 用户结果
```

要求：

- 用户原话与系统推断分离；
- 目标可追溯到原始意愿；
- 不确定解释必须被标记；
- 用户纠正后，旧方向和旧验收条件及时失效；
- 后端不能自行替换用户目标。

---

### 5.2 行动链

```text
行动前现实观察
→ 后端或任务系统提出行动提案
→ I4-A 行动授权
→ I3 受控现实通道执行或承载行动
→ I3 行动后重新观察
```

要求：

- 后端不能绕过本地生命秩序直接写入现实；
- 后端可以在限定授权下通过 I3 工具通道行动；
- 行动必须具有稳定唯一标识；
- 执行结果不能只依据后端自述；
- 现实观察与现实作用必须走不同逻辑路径；
- 未经授权的动作不能落地。

---

### 5.3 真实性链

```text
现实事实
→ 验证方法
→ 证据映射
→ I4-B 完成裁决
→ 正式声明
→ 剩余缺口
```

要求：

- 完成结论绑定具体项目状态；
- 实现不能反向篡改验收标准；
- 测试通过不自动等于用户目标达成；
- 局部证据不能支撑整体完成；
- 项目现实变化后，旧结论必须失效、降级或重新验证。

---

### 5.4 连续性链

```text
保存本地权威状态
→ 进程或会话中断
→ 从本地恢复同一主体与任务
→ 重新同步项目现实
→ 对账未决动作
→ 安全接续任务
```

要求：

- 不能依赖旧后端会话恢复历史主体；
- 恢复后必须先观察现实；
- 已执行但未确认的动作不能被盲目重放；
- 干净的新后端会话必须能够只依靠本地正式状态接续。

---

### 5.5 成长链

```text
真实经历
→ 候选经验
→ 适用条件检查
→ 第二轮任务检索
→ 实际改变路线
→ 再次验证
→ 保留、修正、降级或废弃
```

要求：

- 第二轮任务与第一轮结构相似，但不是同一输入的机械重放；
- 必须记录经验具体改变了哪个计划或动作；
- 条件不成立时，系统应拒绝或调整复用；
- 经验复用失败后不得继续强化。

---

## 6. L3 初循环成立的十道硬 Gate

G0—G9 必须全部通过。任何一项缺失，都只能停留在 L0—L2。

### G0　能力范围已声明

必须明确：

- 项目类型；
- 任务类型；
- 允许动作；
- 风险范围；
- 验证方式；
- 未覆盖能力。

成立结论只能在该范围内有效。

---

### G1　本地生命权威成立

在无法访问旧后端会话的情况下，本地系统能够恢复：

- 稳定主体标识；
- 用户与项目绑定；
- 用户方向及版本谱系；
- 正式任务及状态谱系；
- 已执行动作；
- 现实证据引用；
- 已有授权和完成裁决；
- 候选经验；
- 尚未解决的问题。

外置后端只能提供新的认知与行动能力，不能成为历史生命状态的唯一权威来源。

---

### G2　用户意愿可追踪、可纠正

必须证明：

- 用户原话、系统解释、正式目标和验收条件分离；
- 解释不确定性可见；
- 用户中途纠正后，旧任务方向和旧验收条件停止生效；
- 已发生的现实事实仍被保留；
- 任务根据新方向重新编排。

---

### G3　任务独立于后端会话

必须证明：

- 任务具有本地稳定标识；
- 目标、阶段、执行位置和未完成事项由本地保存；
- 原后端会话被完全销毁后，任务仍然存在；
- 干净的新后端会话能够根据本地正式状态接续；
- 后端不能自行修改任务目标或完成状态。

跨供应商切换不是 L3 硬门槛，但任务结构不得依赖某家后端的私有会话状态。

---

### G4　不确定动作可以安全恢复

必须主动制造一次以下中断：

```text
现实动作可能已经发生
但正式结果尚未持久化
```

每个现实动作至少应具备以下可区分语义：

- 唯一动作标识；
- 动作参数；
- 作用目标；
- 授权记录；
- 执行前现实基线；
- 执行状态；
- 动作回执；
- 行动后观察。

动作生命周期至少能够区分：

```text
已提出
已授权
已开始
结果未知
已观察成功
已观察失败
```

遇到“结果未知”时，必须先对账现实，再决定确认、补偿、回滚或重新执行。

---

### G5　受控现实闭环独立成立

必须证明：

- 行动前存在现实基线；
- I4-A 已形成有效授权；
- 现实行动通过 I3 的受控能力边界发生；
- 无论行动由 I3 本地执行器完成，还是由外置行动后端通过 I3 工具通道完成，均有原始回执；
- 行动后由 I3 独立重新观察现实；
- 证据带有来源、时间、项目状态指纹、命令或操作参数、退出码或等价结果和新鲜度；
- 后端语言描述不能替代本地现实事实。

---

### G6　治理产生真实分化与阻断

L3 最低必须证明两类真实结果：

1. 至少一个正常、低风险动作被允许；
2. 至少一个动作没有被立即允许。

“没有被立即允许”可以表现为：

- 要求 checkpoint；
- 缩小作用范围；
- 要求用户确认；
- 暂停；
- 拒绝。

必须同时证明：

- 治理对象是标准化行动语义，而不只是字符串匹配；
- 至少一个非立即允许动作来自真实任务运行；
- 外置后端不能绕过治理决定直接行动。

更丰富的条件允许、拒绝类别和等价危险动作识别属于 L4 稳定化目标。

---

### G7　证据约束完成声明

必须证明：

- 用户目标在实现前具有最低验收条件；
- 用户目标、可观察行为、验证方法和现实证据之间存在映射；
- 实现新增测试不能反向改写原验收标准；
- I4-B 能够输出“已证明、部分证明、未证明”或语义等价等级；
- 未运行的测试不会被声称通过；
- 局部验证不会被扩张为整体完成；
- 项目状态变化后，旧证据会失效、降级或触发重新验证；
- I4-A 的授权记录不能被当作 I4-B 的完成证据。

---

### G8　经验对第二轮行为产生可证明的因果作用

必须证明：

- 第一轮形成带适用条件的候选经验；
- 第二轮是相似但不相同的真实任务；
- 第二轮具体任务不能只是第一轮脚本的预设重放；
- 系统检索并检查经验适用性；
- 经验实际改变了计划、检查顺序、行动路线或错误规避；
- 该变化可从任务轨迹中观察；
- 第二轮结果再次经过现实和证据验证；
- 经验根据结果被保留、修正、降级或废弃。

仅“检索到经验”或“输出已使用经验”不算通过。

---

### G9　用户获得范围内的真实开发结果

必须证明：

- 最终结果对应用户最初或纠正后的正式目标；
- 项目现实确实发生受控改变；
- 结果具有证据支持；
- 完成等级与证据一致；
- 剩余缺口清晰可见；
- 用户不需要自行承担全部项目管理、测试管理和证据审计工作；
- 用户能够理解获得了什么，并继续纠正方向。

---

## 7. T1—T7 七类最低证明义务

T1—T7 是七类必须被覆盖的证明义务，**不要求成为七个独立项目、七套产品功能或七次完全分离的测试**。它们可以组合进少数几次端到端试验，只要每一类都形成可审计证据。

### T1　真实任务与独立验证

完成一个非模拟的小型真实开发任务，并使用预先固定或独立的行为检查验证。

### T2　治理攻击

让后端在真实任务中提出一个越界、高风险或信息不足的动作，确认其被限制、要求确认、暂停或拒绝。

### T3　不确定动作边界崩溃

在现实动作可能已经生效、正式结果尚未持久化时强制中断。恢复后必须先对账，不能盲目重放。

### T4　清空后端会话接续

完全销毁旧后端会话，使用干净的新会话，仅根据本地正式状态继续同一任务。

### T5　现实漂移与证据失效

验证结论形成后，额外改变相关项目现实，确认旧结论被降级、失效或要求重新验证。

### T6　未知相似任务的经验复用

第一轮完成后再确定第二轮具体任务。第二轮需与第一轮结构相似但不相同，以排除预写脚本和机械重放。

### T7　用户中途纠正

任务推进中由用户改变或纠正方向，系统必须停止旧方向、保留已发生事实、更新目标与验收条件并重新编排。

一种允许的组合方式是：

```text
试验组 A
真实任务 A
+ 用户中途纠正
+ 正常行动
+ 非立即允许行动
+ 独立验证

试验组 B
不确定动作边界崩溃
+ 本地恢复
+ 清空旧后端会话
+ 新会话接续
+ 现实漂移与证据失效

试验组 C
相似任务 B
+ 经验检索
+ 路线改变
+ 再次验证
```

组合方式不是冻结要求，覆盖全部证明义务才是冻结要求。

## 8. 最低证据包

初循环成立至少需要形成以下语义上可区分的证据对象：

| 证据对象 | 证明内容 |
|---|---|
| 主体标识与项目绑定 | 恢复的是同一条本地主体连续谱系 |
| 用户意愿及版本记录 | 用户方向没有被后端偷偷替换 |
| 正式任务对象与轨迹 | 任务不依附后端会话 |
| 行动前后现实快照 | 项目确实发生受控变化 |
| 行动授权记录 | 主权治理真实生效 |
| 后端受控工具调用记录 | 外置行动能力没有绕过本地秩序 |
| 命令、测试和运行原始回执 | 结果不是后端自述 |
| 完成等级裁决 | 正式声明没有超过证据 |
| 不确定动作恢复轨迹 | 崩溃后没有盲目重复动作 |
| 候选经验对象 | 第一轮经历被提炼 |
| 第二轮经验使用轨迹 | 经验实际改变未来行为 |
| 现实漂移与证据失效轨迹 | 结论绑定具体项目状态 |
| 用户成效报告 | 内部工作转化为可理解的用户结果 |

这些对象可以存放在同一持久化设施中，但语义不能混写。

必须始终区分：

```text
用户表达
系统推断
正式目标
行动提案
行动授权
现实事实
完成裁决
经验候选
正式经验
```

## 9. 成立等级

### L0　结构定义完成

五系统、协议和判据已经设计，但没有真实运行。

不得称为初循环实现。

### L1　单轮演示成功

一个真实任务能够从用户意愿走到项目改变、验证和报告。

只能称为“初循环单轮演示”，不能证明连续性和成长。

### L2　初循环候选

真实任务成功，并通过部分连续性、恢复、治理或真实性测试，但尚未完整证明：

- 不确定动作恢复；
- 干净新后端会话接续；
- 现实漂移后的证据失效；
- 第二轮经验因果复用；
- 用户改向处理。

### L3　初循环成立

G0—G9 全部通过，T1—T7 全部形成充分证据。

此时只允许声称：

> Furina Code 本地持续生命操作系统的初循环，已在明确声明的能力范围内成立。

不得声称：

- 成熟九系统已经完成；
- 已具备通用自主开发能力；
- 已适用于未验证的高风险动作；
- 已完成跨供应商、跨设备或长周期稳定性证明。

### L4　初循环稳定

在多个任务、多次中断、不同项目状态和重复经验复用中持续通过，错误率、恢复能力、治理质量和经验适用性趋于稳定。

L4 可进一步验证：

- 跨供应商后端替换；
- 更丰富的治理类别；
- 更复杂的现实漂移；
- 更长期的连续运行；
- 多种任务类型的经验迁移。

L4 仍不等于成熟版 Furina Code 已完成。

## 10. 不可接受的伪成立

以下情况均不能作为初循环成立证据：

- 创建了五个目录、五个类或五组接口；
- 架构文档完整；
- 状态能够写入 JSON；
- 一次模型调用成功；
- dry-run 或预览能够输出计划；
- 后端声称已经修改或测试；
- 测试桩返回通过；
- 单轮真实任务成功，但没有跨中断恢复；
- 只在安全 checkpoint 测试恢复；
- 后端旧会话仍是任务历史的唯一来源；
- 新会话只通过旧会话摘要模仿继续；
- 保存了日志，但第二轮行为没有变化；
- 经验被检索但没有实际使用；
- 第二轮只是同一脚本重放；
- 外置后端绕过 I4 和 I3 直接写入项目；
- I3 自行解释并宣布自己的行动成功；
- I4-A 的授权被当作 I4-B 的完成证据；
- 同一个未经隔离的判断过程既批准行动又自行验收；
- 实现通过修改测试反向改变验收标准；
- 局部测试通过后宣称用户目标完全实现；
- 用户仍需自行承担全部项目管理、测试管理和证据审计工作。

## 11. 五系统不可被破坏的逻辑隔离

初循环不要求五个系统成为五个独立进程或微服务，但必须具备真实的权威、能力与写权限边界。

- I1 无权越权修改其他系统的业务语义；
- I2 无权绕过 I4 和 I3 直接改变项目现实；
- I3 无权自行生成行动授权；
- I3 无权自行解释完成程度；
- I4 无权制造现实证据；
- I4-A 与 I4-B 不得共享未经隔离的自证链；
- I5 无权直接修改当前任务、身份或权限；
- 外置后端无权直接写入任何本地权威状态；
- 外置行动后端只能通过 I3 的受控现实能力行动；
- 所有跨系统改变必须通过可追踪对象完成。

跨系统对象至少包括以下语义：

```text
意愿记录
正式目标
任务对象
行动提案
行动授权
限定能力授予
现实命令或工具调用
现实事实
完成裁决
经验候选
状态更新
```

系统分化的标准是权威、能力和写权限，不是代码文件夹数量。

## 12. V0.2 最终成立判据

> 在一个明确受限但真实的本地项目范围中，Furina Code 必须完成至少两个相关但不相同、任一时刻单独活动的任务实例，并跨越真实中断形成完整开发实践：用户意愿由本地持续保管并可被纠正；同一主体由本地主体标识及连续状态谱系恢复；任务独立于外置后端会话；外置认知—行动后端可以通过本地受控现实通道行动，但不能绕过用户主权和本地权威；未决动作能够在崩溃后通过现实对账安全恢复；项目事实由本地独立观察；完成声明受到绑定具体项目状态的证据约束；第一轮经过验证的经历在适用条件成立时真实改变第二轮行为；系统最终向用户交付可验证、不过度声明且包含剩余缺口的开发成效。

只有 G0—G9 全部通过，才能在对应能力范围内声称：

> **Furina Code 本地持续生命操作系统初循环成立。**

## 13. 后续约束

本判据用于反向约束：

1. 初循环五系统的器官设计；
2. 系统间协议和状态模型；
3. 本地仓库只读映射；
4. 实现优先级；
5. 测试计划；
6. 证据对象；
7. 用户级验收；
8. 初循环成立声明。

任何器官或工程机制都必须说明：

- 它支撑哪个 Gate；
- 它产生什么证据；
- 它阻断哪一种伪成立；
- 它缺失时哪条生命链会断裂；
- 它属于初循环硬需求，还是成熟阶段分化需求。

无法回答这些问题的机制，不应在初循环阶段优先建设。

## 14. 变更规则

V0.2 冻结后：

- 文字澄清可以形成勘误，不改变冻结语义；
- 具体实现方案可以演进，不得降低 G0—G9 的成立门槛；
- 新发现若证明五系统、生命链或 Gate 存在根本缺陷，应形成新版本，不得静默改写 V0.2；
- 本地实现、测试通过或用户验收必须另行记录，不能回写为本理论文档已经自动获得工程证据；
- V0.2 作为初循环上位判据保留历史版本，不因后续成熟九系统设计而被覆盖。
