# 第一部分：总宪法原则

## 2. 类型化权威原则

“某系统拥有某状态”不足以构成完整权威定义。

每个正式对象必须声明以下角色中的适用项：

```text
AuthorityObject
=
对象类型
+ 来源权威
+ 解释权威
+ 提案权威
+ 批准权威
+ 正式状态拥有者
+ 执行者
+ 观察者
+ 验证者
+ 登记者
+ 争议处理者
+ 版本
+ 证据引用
+ 生效条件
+ 失效条件
```

并非每个对象都需要全部角色，也不要求每个角色都由不同物理进程承担。

同一系统可以承担多个角色，但必须同时满足：

- 不形成自我批准；
- 不形成封闭证明环；
- 本文件要求隔离的内部逻辑域确实隔离；
- 状态变化可审计、可重放；
- 高影响对象遵守强制分权；
- 任何角色合并均不得扩大该系统的冻结权力。

不得用一个模糊的“拥有者”替代整条变化链。

---

## 3. 单一正式状态拥有者原则

每一种正式状态必须有且只有一个主要写入权威。

允许：

- 多系统读取；
- 多系统提出提案；
- 多系统提供证据；
- 多系统订阅变化；
- 多系统触发复审。

不允许：

- 多系统同时拥有最终写入权；
- 通过共享数据库字段形成隐性共写；
- 任何系统绕过正式状态转换入口直接改写；
- 恢复脚本无审计修改正式权威状态。

---

## 4. 提案、批准、执行、观察、验证、登记分离原则

以下行为不得在同一个未隔离闭环中合并：

```text
提出变化
批准变化
执行变化
观察结果
验证结果
登记正式状态
```

标准模式：

```text
提案者提出
→ 批准权威批准
→ 执行者执行
→ 独立观察者重新观察
→ 验证者判定
→ 状态拥有者登记
```

任何系统即使在不同阶段参与，也必须依靠内部逻辑域和审计边界保持隔离。

---

## 5. 无封闭证明环原则

任何高影响结论都不得由一组彼此循环引用、且没有外部现实来源或独立验证边界的内部状态自行证明。

至少需要：

```text
一个独立于结论生成过程的现实来源
+
一个不能被被评估流程临时修改的验证规则
```

适用于：

- 行动成功；
- 完成结论；
- 责任关闭；
- 能力成立；
- 经验晋升；
- 用户效果；
- 成熟声明。

禁止：

```text
计划证明行动
行动证明计划
计划与行动共同证明完成
```

---

## 6. 超时不得产生权威转移与幂等原则

等待超时只能改变运行状态，不能自动转移权威。

通用允许结果：

```text
继续等待
暂停
降级
取消提案
重新请求
交还用户
进入安全停止
```

通用禁止结果：

```text
等待授权超时 → 自动授权
等待验证超时 → 自动完成
等待用户超时 → 默认同意
等待观察超时 → 使用推测事实
等待恢复超时 → 让后端接管生命权威
```

不同请求类型的冻结语义如下：

| 请求类型 | 超时后的合法结果 |
|---|---|
| 只读查询 | 可安全重试；必须验证对象版本 |
| 用户确认 | 保持等待、暂停或取消；不得默认同意 |
| 授权请求 | 保持未授权；不得自动放行 |
| 验证请求 | 保持未验证；不得自动完成 |
| 现实动作 | 进入“结果不确定”；先重新观察现实，再决定继续、补偿或重试 |
| 恢复请求 | 保持降级或安全停止 |
| 后端请求 | 可重试或切换，但本地任务、责任与未决动作不得丢失 |

所有可能产生现实副作用的动作必须绑定幂等标识或等价去重机制。

现实动作超时后的固定顺序为：

```text
先观察现实
→ 判断动作是否已经发生、部分发生或未发生
→ 再决定继续、补偿、取消或重试
```

不得在结果不确定时盲目重复现实动作。

## 7. 用户主权决定与执行授权分离原则

用户决定表达的是方向、价值与主权。

执行授权表达的是：

- 具体作用对象；
- 作用范围；
- 风险；
- 有效期；
- 可逆性；
- 执行次数；
- 预算；
- 撤销条件。

因此：

```text
用户主权决定
≠
可执行授权票据
```

标准链条：

```text
S2确认用户主权语义
→ S6核对身份、范围、影响和知情确认
→ S6签发有限授权票据
→ S3 / S9执行
```

---

## 8. 失败时权力收缩原则

系统失效、冲突、不确定或恢复中，默认行为必须是：

```text
减少权限
缩小范围
保护现场
保留证据
停止不可逆动作
等待对账、恢复或用户决定
```

不得因为失败：

- 扩大权限；
- 跳过验证；
- 删除失败证据；
- 自动接管其他系统权威；
- 将未知状态写为成功；
- 让后端推测替代现实。

---

## 9. 真实性高于效率、稳定性与成长

以下理由均不能降低真实性：

- 成本不足；
- 后端能力下降；
- 系统希望维持成熟；
- 经验表明某条路线常常成功；
- 用户希望低打扰；
- 系统希望避免回滚。

当证据不足时，必须：

- 缩小声明；
- 标记未知；
- 重新验证；
- 降级；
- 暂停；
- 交还用户判断。

---

## 9.1 单调保守转换原则

当权威系统、裁决边界或验证器暂时不可用时，系统不得因为等待恢复而维持已经失真的高信任状态。

冻结规则是：

> **降低信任、缩小权力、暂停声明的转换，可以依据预先冻结的失效规则自动发生；提高信任、扩大范围、恢复声明，必须经过原批准权威重新裁决。**

适用示例：

```text
成熟有效 → 暂停待复审
能力有效 → 待重验 / 退化 / 未知
授权有效 → 暂停 / 撤销 / 过期
完成有效 → 待重验 / 失效
程序经验有效 → 降级 / 停用
后端可用 → 不可用 / 未知
```

反向转换不得自动发生：

```text
暂停 → 有效
未知 → 已验证
撤销 → 恢复
退化 → 完整能力
经验停用 → 正式程序经验
```

保守转换必须：

- 由冻结失效规则触发；
- 记录触发事实和对象版本；
- 不删除历史状态；
- 对用户和依赖系统可见；
- 在恢复后进入重新批准或重新验证流程。

# 第三部分：S1 主体身份与连续性系统

## 11. S1 的唯一生命责任

> 维护 Furina Code 的主体身份、连续谱系、核心定义版本与合法变化历史。

---

## 12. S1 的类型化权威

### 来源权威

- 用户身份与项目绑定的原始来源：用户、可信恢复事实或正式迁移事实；
- 核心宪法来源：冻结文档及正式变更程序；
- 成熟声明来源：C1。

### 提案权威

可由：

- S2 提交用户绑定变化提案；
- S9 提交恢复或迁移提案；
- S8 提交高影响长期行为变化提案；
- 正式变更程序提交核心定义变化提案。

### 批准权威

S1 不拥有身份变化批准权。

批准必须来自：

- 用户明确确认；
- 正式恢复或迁移规则；
- 最高级宪法变更程序；
- C1 对成熟声明的裁决。

### 正式状态拥有者

S1拥有：

- 主体标识；
- 用户—主体绑定；
- 项目—主体绑定；
- 主体连续谱系；
- 核心定义版本；
- 宪法版本引用；
- 正当演化历史；
- 主体恢复锚点；
- 主体终止状态；
- 成熟声明本地登记状态。

### 登记权威

S1负责登记，但登记不赋予合法性。

S1只能登记已经完成合法批准的变化。

### 争议处理

- 用户绑定争议：回到用户与 S2；
- 恢复连续性争议：S9 提供恢复事实，S7验证证据；
- 成熟声明争议：C1；
- 核心定义争议：最高级变更程序。

---

## 13. S1 禁止权力

S1不得：

- 自行批准身份变化；
- 自行解释用户方向；
- 吞并任务、事实、证据、经验与恢复状态；
- 自行签发成熟声明；
- 因连续性需要拒绝用户终止；
- 以登记结果反向证明变化合法。

---

## 14. S1 失效姿态

当主体谱系不可信：

- 停止建立新正式责任；
- 停止恢复成熟声明；
- 禁止后端会话充当临时身份源；
- 保留只读导出与诊断；
- 由 S9 启动恢复；
- 恢复前不得声称主体连续。

# 第四部分：S2 用户关系与意愿承接系统

## 15. S2 的唯一生命责任

> 保存并解释用户意愿，维护方向谱系，确保系统承接而不篡夺用户方向。

---

## 16. S2 意愿状态与运行假设分层

必须区分：

```text
原始表达
解释候选
待确认解释
正式有效方向
临时授权意愿
运行假设 OperationalAssumption
被拒绝解释
已撤销方向
历史方向
```

### 原始表达

只记录用户实际表达，不得改写。

### 解释候选

由 S2 或外置后端形成，但尚未具有方向权威。

### 待确认解释

需要用户确认才能进入正式方向。

### 正式有效方向

来源只能是：

- 用户明确确认；
- 用户预先明确建立且当前仍有效的持续方向规则。

正式方向不得由低风险过程推测、用户沉默、历史偏好或后端常识自动生成。

### 临时授权意愿

只表达用户愿意进行某类行为，不等于 S6 的可执行授权票据。

### 运行假设 `OperationalAssumption`

运行假设只用于填补不改变方向的临时过程细节，例如读取顺序、低风险检查路径或可撤销的执行顺序。

运行假设必须：

- 不扩大用户目标；
- 不扩大权限；
- 不增加不可逆风险；
- 不改变成本边界；
- 不修改验收标准；
- 可撤销；
- 绑定作用范围和有效期；
- 在现实、用户方向或政策冲突时立即失效；
- 不写入正式方向谱系。

固定关系：

```text
用户方向：S2正式权威
运行假设：临时过程对象
```

## 17. S2 的类型化权威

### 来源权威

用户是方向与价值的最终来源权威。

### 解释权威

S2负责形成解释候选并保留版本。

### 批准权威

用户批准正式方向。

### 正式状态拥有者

S2拥有：

- 用户原始表达；
- 意愿解释版本；
- 当前正式方向；
- 目标语义；
- 验收语义；
- 用户纠正；
- 用户确认、拒绝与撤销；
- 价值冲突；
- 主权决策请求；
- 用户托付体验与主观反馈。

## 30. 普通责任关闭协议


- 删除未决事项；
- 改写关闭条件；
- 缩小责任范围；
- 将持续维护责任改为一次性完成；
- 将用户未确认事项视为已接受；
- 使用过期或版本不匹配的关闭资格。

未决事项必须先分别进入：

- 已完成；
- 持续维护责任；
- 正式移交；
- 用户明确放弃；
- 超出范围；
- 终止收束。

完成这些转换后，才能重新申请关闭资格。

其中：

- S7不能写责任关闭状态；
- S5不能生成完成真实性；
- S2不能代替技术验证；
- C1不参与普通关闭；
- 责任关闭不得由 S5—S7 彼此循环引用自行证明。

C1只在成熟验证时审查这套关闭机制是否可靠。

## 61. 恢复、事务与事件基础设施的权威限制

### 61.2 事务协调器

- 收集各权威系统已经批准的 `TransitionIntent`；

- 修改批准条件；

- 决定冲突语义；
- 在部分失败后自行选择“最合理状态”；
- 成为隐性第十系统。

### 61.3 事件总线

事件只是通知，不是权威状态。

接收方在高影响行动前必须验证：

- 事件来源；
- 对象版本；
- 当前权威状态；
- 事件是否过期；
- 事件是否已撤销或被新版本取代。

不得因为收到一条事件，就绕过权威系统直接行动。

## 68. 用户终止与数据处置协议


终止不得写成“已完成”。

### 68.2 数据处置

用户可以分别选择：

- 本地保留；
- 导出后保留；
- 导出后删除；
- 选择性删除；
- 完整删除。

完整删除前必须明确告知：

- 主体连续性将无法恢复；
- 历史责任链可能无法继续审计；
- 成熟裁决和证据将失去本地依托；
- 删除后系统不得秘密保留副本。

用户知情确认后，系统必须服从完整删除。

固定关系：

```text
终止 ≠ 删除
责任收束 ≠ 永久保留用户数据
```

若选择完整删除，可以先导出最终责任收束清单，随后本地生命权威一并消失。

```text
S7产生真实性与能力事实
→ 命中冻结失效规则时，S1可将本地投影保守转换为“暂停待复审”
→ G11形成成熟状态变化候选
→ C1独立裁决
→ C1签发新的MaturityVerdict
→ S1登记新裁决并更新本地投影
→ S2向用户报告
```

保守暂停不等于 C1 已作出最终撤销裁决。  
从暂停、缩小或过期恢复为有效，必须由 C1 重新签发。

# 第十六部分：冲突裁决规则

S7提供事实，C1签发不可变裁决制品，S1登记引用并维护本地投影。

# 第十八部分：系统劫持禁止条款

## 82. S1 身份劫持

禁止以连续性名义吞并所有长期状态。

## 83. S2 意愿劫持

禁止把推测偏好升级为正式方向。

## 84. S3 现实劫持

禁止把执行回执直接作为最终现实。

## 85. S4 后端劫持

禁止让后端会话成为主体、任务、责任和经验来源。

## 86. S5 任务劫持

禁止自行授权、自行验证、自行关闭。

## 87. S6 治理劫持

禁止以安全名义夺取用户方向和终止权。

## 88. S7 真相劫持

禁止成为不可复核的最高法院。

## 89. S8 经验劫持

禁止历史经验覆盖现实、权限和用户方向。

## 90. S9 稳态劫持

禁止以资源或稳定性为由删除证据、降低安全或抵抗终止。

## 91. C1 裁决劫持

禁止 C1 接管普通任务、授权、现实行动和主体终止。

任一劫持发生，相关成熟 Gate 自动失败。

## 93. 共享组织宪法

共享组织必须：

- 为每个对象标记正式权威系统；
- 禁止非权威系统直接写入；
- 所有状态转换可审计；
- 保持对象版本；
- 保持来源和证据引用；
- 支持只读重放；
- 不因物理共存而混淆语义权威；
- 不得成为恢复后门；
- 不得绕过授权与验证协议；
- 事务协调器只提交已批准的 `TransitionIntent`；
- 事件总线只传递通知，不形成正式状态；
- 共享缓存不得成为现实、证据或授权的最终来源；
- 管理工具和维护脚本必须遵守同一权威链。

## 96. 后续器官设计约束

任何后续器官候选必须回答：

- 它支撑哪些 G 与 C；
- 它缺失时哪条责任链断裂。

## 97. 变更规则

V0.2 冻结后：

- 文字澄清可以形成勘误，但不得改变冻结语义；
- 器官和代码实现可以演进，但不得改变九系统权威边界；
- 数据结构可以变化，但不得绕过类型化权威链；
- 任何降低分权、真实性、用户主权、保守降级或 C1 独立性的修改，必须形成新版本；
- 新发现若证明本宪法存在根缺陷，必须发布 V0.3 或更高版本，不得静默修改 V0.2；
- 本文件必须作为历史冻结锚点长期保留；
- 工程实现、测试通过和成熟成立必须另行提供证据，不得由本文件自动推导。
