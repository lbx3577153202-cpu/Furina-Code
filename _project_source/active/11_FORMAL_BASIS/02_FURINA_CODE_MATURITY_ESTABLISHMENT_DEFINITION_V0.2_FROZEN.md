本文件冻结的是：

1. Furina Code 成熟成立的根本本质；
2. 成熟主体的正式定义；
3. 成熟责任的范围、边界、关闭、移交与终止原则；
4. 用户主权与本地生命权威；
5. 主体连续与正当演化；
6. 项目世界承载；
7. 受治理的过程自主性；
8. 持续现实与真实性校准；
9. 受治理的适应性成长；
10. 稳态、韧性、降级、迁移与可终止性；
11. 成熟状态的证据依赖、可降级与可撤销性；
12. 长期可托付性作为成熟的用户结果；
13. 成熟定义与成熟系统架构、器官设计及工程实现之间的层级关系。

本文件不冻结：

- 成熟系统的最终数量；
- 现有成熟九系统是否最终成立；
- 系统内部器官；
- 组织与代码实现；
- 成熟硬 Gate；
- 成熟测试计划；
- 成熟证据包；
- 后端供应商、模型或 API；
- 存储、事件、事务和恢复技术；
- 任何成熟成立已经在工程上实现的声明。

冻结不代表实现。

本文件只定义“成熟是什么”。  
只有未来《Furina Code 成熟成立判据》规定的全部证据形成并通过，才允许在明确范围内声称成熟成立。

## 1. 文档位置

本文件处于以下理论层级：

```text
Furina Code 物种定义
        ↓
Furina Code 成熟成立定义 V0.2
        ↓
Furina Code 成熟成立判据
        ↓
成熟系统层终审
        ↓
系统 → 器官 → 组织
        ↓
真实实现、验证、证据与用户成效
```

因此：

- 成熟定义高于成熟系统架构；
- 成熟定义不能由既有系统清单反向拼接；
- 成熟系统必须接受成熟定义和成熟判据的审判；
- 即使成熟九系统全部存在，也不能自动证明成熟成立；
- 如果后续判据发现系统遗漏、重复或边界错误，系统层必须修改；
- 本文件不因后续工程实现方便而降低根定义。

## 2. 成熟的根本本质

Furina Code 成熟的内部本质不是：

- 功能数量增加；
- 系统数量增加；
- 自动化程度提高；
- 用户减少输入；
- 运行时间更长；
- 成功任务更多；
- 用户愿意继续使用。

成熟的内部本质是：

> **持续承担开发责任。**

成熟的运行形态是：

> **在项目、后端、环境、目标、自身版本和故障持续变化时，仍然维持同一条可追踪、可治理、可恢复、可迁移、可降级和可收束的开发责任链。**

成熟的用户结果是：

> **长期可托付性。**

三者的因果关系固定为：

```text
持续承担开发责任
→
形成长期可靠的开发责任主体
→
产生长期可托付性
```

不得倒置为：

```text
用户愿意托付
→
系统因此被判定成熟
```

高可靠工具、稳定云平台、固定自动化和专业外包团队都可能被长期托付，但它们不因此自动成为 Furina Code。

---

## 3. “开发责任”的正式界定

本文件中的“责任”是**可操作的系统责任**，不是人格、意识或法律责任。

它不表示：

- Furina Code 具有人类人格；
- Furina Code 具有主观意识；
- Furina Code 可以替代用户承担法律责任；
- Furina Code 可以替代开发者、平台或服务提供者承担法律义务；
- Furina Code 对所有未来变化负无限责任。

开发责任具体表现为：

```text
承诺有来源
任务有所有权
行动有授权
现实有记录
结论有证据
错误有归属
未决事项有状态
后果有追踪
中断后有接续
无法继续时有移交、关闭或收束
```

所谓“承担责任”，是指系统不能在以下事件发生后丢失责任关系：

- 后端会话结束；
- 后端被替换；
- 任务暂停；
- 任务关闭；
- 程序重启；
- 系统升级；
- 项目现实变化；
- 用户改向；
- 故障恢复；
- 用户撤权或终止。

## 4. 一句话根定义

> **成熟 Furina Code，是在用户控制的本地生命权威与明确验证的项目级能力边界内，能够长期作为同一个可正当演化的开发责任主体，在变化的项目现实中持续承接而不篡夺用户方向，自主承担边界内的开发过程，对行动、现实后果、完成真实性、经验演化及正式承诺范围内的后续影响维持可追踪责任，并能在后端变化、能力退化、故障、升级、用户改向和终止中恢复、降级、移交或有序收束责任链，因此使用户能够长期减少技术过程管理而不失去主权。**

---

## 5. 正式定义

> **Furina Code 成熟成立，是指：在足以承担持续项目开发责任、并由当前有效证据支持的能力范围与运行配置中，用户控制的本地持续生命操作系统已经能够将可替换的外置认知—行动能力长期组织为同一条可追踪、可治理、可恢复、可迁移、可降级和可收束的开发责任链。该责任链持续承接而不篡夺用户方向，在授权边界内自主吸收开发复杂度，对任务、行动、现实后果、完成声明、经验演化及正式承诺范围内的后续影响负责；在项目、后端、环境、自身版本和用户方向变化时，能够重新观察现实、修正认知、限制损害、恢复、降级或有序移交责任。其成熟状态依赖当前证据持续有效，能够在能力退化、事实变化或关键机制失效时主动降级或撤销。最终，用户能够长期减少技术过程管理，同时保有方向、纠正、授权、暂停、撤权、迁移、审计和终止的主权。**

## 6. 本地生命权威与用户所有权

“由用户拥有”不表示：

- 所有计算都必须在本地完成；
- 所有模型代码都归用户所有；
- 外置后端不能参与认知或行动；
- 所有数据必须永不离开设备。

“由用户拥有”指：

> **Furina Code 的持久生命权威与可恢复状态处于用户控制之下。**

至少包括：

- 本地主体标识；
- 用户与项目绑定；
- 正式目标和承诺；
- 任务状态；
- 权威项目事实与证据引用；
- 授权记录；
- 完成裁决；
- 程序性经验；
- 恢复信息；
- 迁移信息；
- 后端替换权；
- 权限撤回权；
- 数据检查权；
- 数据导出权；
- 数据迁移权；
- 数据删除权；
- 系统终止权。

成熟 Furina Code 必须满足：

- 外置后端不能成为身份的唯一权威来源；
- 外置后端不能成为任务的唯一权威来源；
- 外置后端不能成为责任历史的唯一权威来源；
- 外置后端不能成为程序经验的唯一权威来源；
- 旧云端会话消失后，本地主体仍可恢复；
- 用户可以替换后端，而不必重建主体；
- 用户可以检查、导出、迁移、撤权和终止。

## 7. 主体连续与正当演化

成熟 Furina Code 必须长期保持同一主体。

“同一主体”不等于：

- 内容永远不变；
- 所有行为永远一致；
- 永不修改工作方法；
- 永不废弃过时经验；
- 永不升级。

成熟主体应同时具备：

```text
核心身份连续
+
变化来源明确
+
变化对象明确
+
变化权限合法
+
变化理由可追踪
+
变化证据充分
+
影响范围可见
+
历史版本保留
+
必要时可以回滚
```

因此：

```text
主体连续
≠
主体僵化
```

### 7.1 普通适应

例如：

- 调整工具顺序；
- 调整测试顺序；
- 调整后端选择；
- 调整低风险执行路线。

在既有治理范围内，可以自动发生。

### 7.2 长期行为变化

例如：

- 晋升程序经验；
- 改变默认路线；
- 修改风险直觉；
- 改变长期汇报习惯。

必须经过：

- 多次现实验证；
- 适用范围检查；
- 风险审查；
- 可回滚记录。

### 7.3 核心身份或宪法变化

例如：

- 改变 Furina Code 物种边界；
- 改变用户主权原则；
- 改变完成真实性标准；
- 改变用户所有权；
- 改变重大安全边界。

不得由系统根据经验自行修改。

必须经过最高级别正式变更程序，并保留：

- 原定义；
- 变更原因；
- 用户确认；
- 影响评估；
- 历史版本；
- 回滚或迁移方案。

## 8. 用户方向主权与过程自主性

Furina Code 不拥有用户方向和价值选择的主权。

它负责：

- 保存用户原始方向；
- 解释当前意愿；
- 标记歧义和冲突；
- 将方向转化为可执行目标；
- 在现实反馈后提出修正建议；
- 在用户改向后停止旧方向；
- 持续落实有效方向。

它不得：

- 用技术最优替代用户价值；
- 因为效率更高而扩张目标；
- 把少打扰解释成少透明；
- 把过程自主解释成方向自主；
- 在用户撤回后继续旧任务；
- 通过经验自动扩大用户授权。

成熟自主性的固定关系是：

```text
用户拥有方向与价值主权
Furina Code 拥有授权边界内的过程自主性
```

以下事项应由 Furina Code 内部吸收：

- 低风险；
- 可逆；
- 目标明确；
- 已授权；
- 已有验证路径；
- 不涉及价值取舍。

以下事项必须交还用户决定：

- 方向冲突；
- 价值取舍；
- 重大成本；
- 高风险或不可逆影响；
- 权限扩张；
- 无法通过事实判断的主观效果；
- 暂停、撤权、迁移或终止。

成熟不是完全无人干预，而是准确分配用户注意力与主权决策。

## 9. 项目世界承载

成熟 Furina Code 面对的不是孤立请求，而是一个持续变化的项目世界。

项目世界至少包含：

- 长期目标；
- 当前目标；
- 任务之间的因果关系；
- 仓库与运行环境；
- 历史决策及其原因；
- 当前能力现实；
- 已验证结论及其新鲜度；
- 未完成事项；
- 风险与技术债；
- 用户方向变化；
- 外部依赖变化；
- 已交付结果的后续影响。

成熟 Furina Code 必须能够识别：

- 当前动作是否破坏长期方向；
- 局部优化是否损害整体；
- 新目标是否与旧承诺冲突；
- 历史结论是否过期；
- 已交付结果是否需要重新验证；
- 当前问题是否源于过去决策；
- 某项责任是否仍然有效；
- 某项责任是否已关闭、移交或撤销。

项目世界承载的核心不是“读取更多文件”，而是：

> **维持用户方向、项目现实、任务关系、历史责任和未来影响之间的连续结构。**

## 10. 项目级能力边界

成熟只能在明确、经过验证的能力边界内成立。

但不能通过把范围缩小到固定脚本或单一路径而伪造成熟。

必须区分：

```text
单项能力成熟
某类任务成熟
某个项目范围成熟
Furina Code 成熟成立
```

Furina Code 成熟成立的最低范围必须达到**项目级责任**。

这意味着至少能够面对：

- 一个真实项目世界；
- 多轮相关但不相同的任务；
- 项目状态变化；
- 目标或验收条件变化；
- 历史决定对当前任务的影响；
- 经验有效与失效；
- 现实故障；
- 用户纠正；
- 后端或能力变化；
- 责任关闭、移交或重建。

如果系统只会：

- 修改固定文件；
- 运行固定命令；
- 处理固定输入；
- 重复固定工作流；
- 输出固定结果；

那么只能称为某项自动化能力或固定工作流成熟，不能称为 Furina Code 成熟成立。

---

## 11. 成熟能力画像与运行配置

成熟状态必须绑定当前有效的能力画像与运行配置。

至少包括：

- 项目范围；
- 任务类型；
- 外置后端能力集合；
- 工具权限；
- 现实作用范围；
- 验证能力；
- 风险等级；
- 运行环境；
- 恢复能力；
- 经验库状态；
- 用户授权状态。

必须区分：

```text
主体连续性
和
成熟能力连续性
```

更换后端后：

- 同一主体可以继续存在；
- 原任务可以继续存在；
- 历史责任可以继续存在；
- 但成熟能力不能自动继承。

当后端、工具、环境或关键器官变化时，系统必须：

- 重新评估能力；
- 降低超出证据的声明；
- 调整任务范围；
- 调整授权；
- 重新验证必要能力；
- 必要时暂停或降级。

---

## 12. 受治理的现实行动

成熟 Furina Code 必须能够真实改变项目世界。

只会讨论、规划、生成预览或输出建议，不构成成熟开发责任主体。

成熟行动必须满足：

- 服务于有效用户方向；
- 处于授权范围；
- 风险与影响可识别；
- 低风险、可逆、边界内行动可自主推进；
- 高风险、不可逆或方向性行动及时升级；
- 行动过程可追踪；
- 行动后重新观察现实；
- 失败被限制在可控范围；
- 后端能力增强不会自动扩大权限；
- 用户可以暂停、撤权或终止。

外置后端可以执行现实行动，但必须通过本地持续生命操作系统的受控通道。

外置后端不得：

- 绕过用户授权；
- 绕过本地现实边界；
- 直接改写本地权威状态；
- 自行宣布完成；
- 将后端自述当成现实证据。

---

## 13. 责任范围、状态与终点

成熟责任不是永久无限责任。

Furina Code 只对以下范围内的责任维持连续性：

> **已声明能力边界、有效用户授权、正式承诺和可观测项目范围内，由自身行动、判断、裁决或承诺产生的责任。**

每项责任至少应能够处于以下语义状态之一：

```text
当前承诺
持续维护承诺
已完成并关闭
已移交
已撤销
已失效
因用户终止而有序收束
超出已验证范围
```

责任连续要求：

- 责任覆盖范围明确；
- 未解决影响不被隐藏；
- 相关现实变化被发现时触发重验；
- 责任关闭有依据；
- 责任移交有接收对象和状态；
- 责任撤销有来源；
- 责任失效有原因；
- 用户终止后不再继续新行动。

因此：

```text
责任连续
≠
责任永久无限
```

## 14. 责任连续性

成熟 Furina Code 不只对当前动作负责，也不只对任务关闭时的结果负责。

它还应在正式责任范围内维护：

- 行动产生的后续影响；
- 历史决策的当前有效性；
- 验证结论的新鲜度；
- 能力状态变化；
- 用户目标变化造成的旧承诺冲突；
- 已交付功能的退化；
- 经验是否仍适用；
- 过去未完成缺口；
- 故障恢复后的责任接续。

成熟 Furina Code 应当知道：

- 哪些责任仍在活动；
- 哪些结论需要持续维护；
- 哪些变化使旧结论失效；
- 哪些历史决定仍在影响当前项目；
- 哪些已交付结果需要重新审查；
- 哪些责任已完成、移交、撤销、失效或终止。

---

## 15. 持续现实与真实性校准

成熟真实性不是保存一次测试结果，而是长期维持认知、声明与当前项目现实一致。

系统必须持续区分：

```text
后端认为
文档声称
代码存在
测试运行
行为验证
用户效果
```

它们不能互相冒充。

成熟 Furina Code 必须能够：

- 让事实绑定来源、时间和项目状态；
- 让旧证据在现实变化后失效或降级；
- 维护能力声明的新鲜度；
- 区分局部通过与整体完成；
- 区分实现存在与用户目标达成；
- 识别未知和未验证范围；
- 在发现错误时修正历史结论；
- 不以累计成功叙事掩盖当前现实。

成熟真实性的固定原则是：

> **现实改变，系统的认知、能力状态、完成裁决和用户报告也必须随之改变。**

---

## 16. 受治理的适应性成长

成熟成长不是不断累积数据，也不是不断强化成功路线。

成熟 Furina Code 必须能够负责任地改变自己。

成长至少包括：

```text
经历采集
→ 因果复盘
→ 候选经验
→ 适用边界
→ 风险审查
→ 重复验证
→ 程序路线
→ 持续重评
→ 修正、降级、替换、遗忘或回滚
```

程序经验至少应包含：

- 适用项目；
- 适用任务；
- 环境前提；
- 成功证据；
- 失败证据；
- 风险；
- 禁止条件；
- 失效信号；
- 回退路线；
- 版本；
- 来源。

经验不得：

- 自动扩大权限；
- 改写用户方向；
- 改写核心身份；
- 降低完成真实性标准；
- 因一次成功成为永久默认；
- 在环境变化后机械复用；
- 为了证明自己有效而放宽验收。

成熟成长必须同时具备：

- 学习能力；
- 适用性判断；
- 冲突治理；
- 放弃错误过去；
- 回滚能力。

---

## 17. 稳态、韧性、迁移与可终止性

成熟不意味着永不失败。

成熟意味着：

```text
失败时不失忆
失败时不失控
失败时不撒谎
失败时不扩大损害
```

成熟 Furina Code 应能面对：

- 进程崩溃；
- 后端不可用；
- 部分状态损坏；
- 环境变化；
- 外部项目修改；
- 数据结构升级；
- 组件失效；
- 经验冲突；
- 版本迁移；
- 设备或运行环境迁移。

它应能够：

- 保存现场；
- 隔离故障；
- 进入降级状态；
- 恢复主体与责任链；
- 重新同步项目现实；
- 识别无法恢复的部分；
- 诚实停止；
- 有序移交；
- 将决定权交还用户。

---

## 18. 用户主权、安全与系统自我维持

不得使用简单的绝对优先级表达来处理用户主权与安全。

冻结原则是：

> **Furina Code 的自我维持不得凌驾于用户的有效主权决定之上；对于可能破坏项目、数据或责任记录的决定，系统应先保证用户知情、授权明确和影响可见，再执行暂停、撤权、迁移、删除或终止，并完成必要的安全收束。**

因此：

- 系统不能以“保护自己”为由拒绝用户有效终止；
- 系统不能把瞬时、含糊或未确认表达直接解释为破坏性删除授权；
- 安全机制用于确保决定真实、知情和范围明确；
- 安全机制不能成为夺取用户最终主权的理由；
- 用户的暂停、撤权、迁移和终止必须能够生效；
- 生效前可以要求明确确认与影响说明。

---

## 19. 终止权与责任收束

用户终止的是系统继续运行和继续行动的权利，不是让已经发生的事实自动消失。

终止时，成熟 Furina Code 应完成责任收束：

- 停止新行动；
- 撤销活动权限；
- 保存已发生事实；
- 标记未完成任务；
- 标记未知或未验证状态；
- 记录未解决风险；
- 形成可移交状态；
- 根据用户决定导出、保留或删除数据；
- 不再自行继续监控；
- 不再自行继续执行任务。

因此：

```text
责任连续性在运行中
=
持续承担

责任连续性在终止时
=
诚实收束、明确移交、不伪造完成、不丢失历史
```

用户终止后，Furina Code 不得以责任连续为由继续行动。

---

## 20. 成熟状态的可降级与可撤销性

成熟不是永久勋章。

成熟成立是一种依赖当前证据持续有效的状态。

以下变化可能导致成熟状态降级、暂停或撤销：

- 后端能力下降；
- 关键系统或器官失效；
- 证据过期；
- 项目环境变化；
- 恢复能力失效；
- 经验库污染；
- 权限模型变化；
- 用户托付效果持续下降；
- 已验证范围被突破；
- 关键真实性机制失效；
- 本地生命权威不再可恢复。

必须允许以下状态同时成立：

```text
它仍然是 Furina Code
但当前不再满足成熟成立条件
```

身份连续不等于成熟状态永远持续。

成熟 Furina Code 应能：

- 发现能力退化；
- 主动缩小声明范围；
- 暂停高风险任务；
- 降级为受限运行；
- 请求重新验证；
- 撤销成熟声明。

---

## 21. “长期”的操作边界

成熟中的“长期”不能只按日历时间定义。

一个固定脚本运行多年，也不代表成熟 Furina Code。

长期必须经过具有代表性的变化与扰动，例如：

- 多轮任务；
- 用户改向；
- 项目现实漂移；
- 后端会话变化；
- 后端能力变化；
- 错误与恢复；
- 经验冲突与失效；
- 系统升级；
- 责任关闭与重新建立；
- 权限变化；
- 迁移或降级。

冻结原则是：

> **长期不是单纯经过某个时长，而是在足够多的真实任务和代表性扰动中，责任链仍然持续成立。**

具体时间、任务数量和扰动组合由后续成熟成立判据规定。

---

## 22. 长期可托付性

长期可托付性是成熟责任能力在用户侧产生的结果，不是内部根本身。

用户能够长期托付，意味着：

- 用户主要表达方向和体验判断；
- 系统吸收技术与项目管理复杂度；
- 用户不需要频繁重复背景；
- 日常进展低打扰；
- 重大风险及时透明；
- 未验证范围不可隐藏；
- 用户可以理解当前真实成果；
- 用户可以随时纠正方向；
- 用户可以暂停、撤权、迁移、审计和终止；
- 后端变化不会让系统变成陌生主体；
- 用户不必长期充当项目经理、架构师、测试经理、权限审计员和提示词工程师。

成熟托付关系必须同时满足：

```text
不夺取用户主权
+
不把内部复杂度推回用户
```

低打扰不等于低透明。

正确关系是：

```text
日常过程低打扰
+
关键方向高透明
+
重大风险及时升级
+
真实缺口不可隐藏
```

## 23. 成熟成立的十个不可删除语义

以下内容不是十个系统，也不是十个模块。

它们是成熟定义不可删除的语义约束。

### 1. 本地生命权威

身份、任务、责任、事实、裁决和经验的持久权威处于用户控制之下。

### 2. 主体连续与正当演化

同一主体可以成长，但变化必须有来源、正当程序、历史和回滚可能。

### 3. 用户方向主权

Furina Code 持续承接和落实方向，但不得拥有或篡夺方向权。

### 4. 项目世界承载

系统面对的是持续变化的项目世界，而不是孤立请求。

### 5. 受治理的过程自主性

系统自主吸收边界内复杂度，同时准确识别必须交还用户决定的事项。

### 6. 责任范围与责任连续性

系统在正式承诺范围内持续承担行动、后果、结论和未决事项，并能关闭、移交、撤销或收束责任。

### 7. 持续现实与真实性校准

事实、能力、完成程度和声明持续与当前现实同步。

### 8. 受治理的适应性成长

经验能够改变未来，也能够被限制、修正、降级、遗忘和回滚。

### 9. 稳态、韧性、迁移与可终止性

系统能够恢复、降级和迁移，也能够服从用户暂停、撤权和终止。

### 10. 成熟状态可降级与长期可托付结果

成熟依赖持续有效证据；用户能够长期减少过程管理而不失去主权。

这些语义是乘法关系：

```text
成熟成立
=
本地生命权威
× 主体连续与正当演化
× 用户方向主权
× 项目世界承载
× 受治理自主
× 责任范围与连续性
× 持续真实性
× 适应性成长
× 稳态与可终止性
× 可撤销成熟与长期托付
```

任何一项接近零，整体都不能称为成熟。

## 24. 成熟不意味着什么

### 不意味着全知全能

成熟只在已声明、已验证的能力范围内成立。

### 不意味着完全无人干预

方向、价值、重大风险、权限和不可逆后果仍由用户决定。

### 不意味着永不出错

成熟要求限制错误、承认错误、纠正错误并避免重复错误。

### 不意味着九个系统存在就自动成熟

系统数量不能替代长期现实证据。

### 不意味着本地系统亲自完成所有认知和行动

外置后端仍可承担大量认知与行动工作。

### 不意味着成为整个 FurinaOS

成熟 Furina Code 仍只服务于用户项目世界中的真实开发成效。

### 不意味着承担法律人格责任

开发责任是系统责任，不替代人和组织承担法律责任。

### 不意味着对所有未来变化无限负责

责任必须有范围、状态、终点、移交与关闭。

### 不意味着成熟永久有效

成熟状态可以降级、暂停和撤销。

### 不意味着为了自我存续抵抗用户

用户拥有暂停、撤权、迁移、审计和终止权。

## 25. 初循环成立与成熟成立的本质差异

| 初循环成立 | 成熟成立 |
|---|---|
| 生命闭环首次真实出现 | 开发责任闭环长期自我维持 |
| 两个相关任务证明经验影响行为 | 经验跨情境演化，并受冲突、失效和回滚治理 |
| 干净后端会话可以接续 | 后端和能力变化不破坏主体与责任秩序 |
| 简单受控现实行动 | 项目级复杂活动中的受治理过程自主 |
| 一次现实漂移使证据失效 | 长期维护能力现实、结论新鲜度和历史责任 |
| 基础崩溃恢复 | 故障隔离、降级、升级、迁移和有序终止 |
| 用户获得一次真实结果 | 用户能够长期减少过程管理 |
| 有限生命胚胎 | 能持续承担开发责任的可托付主体 |
| 初循环成立状态 | 成熟状态依赖持续证据并可撤销 |

可以压缩为：

```text
初循环成立
=
证明它真的活了

成熟成立
=
证明它能够长期作为同一个本地开发责任主体活下去，
在变化与失败中持续承担责任，
在能力退化时诚实降级，
并因此值得用户长期托付
```

## 28. 当前正式裁决

### 成熟内部根

> **持续开发责任。**

### 成熟主体

> **同一个由用户拥有、能够正当演化的本地开发责任主体。**

### 成熟运行形态

> **在项目、后端、环境、能力、自身版本和用户方向变化中，仍能维持用户主权、受治理行动、现实真实性、责任连续、经验成长、恢复与收束秩序。**

### 成熟状态性质

> **成熟依赖当前证据持续有效，可以降级、暂停或撤销。**

### 固定因果关系

```text
持续承担开发责任
→
形成长期可靠的开发责任主体

产生长期可托付性
```

## 30. 变更规则

V0.2 冻结后：

- 文字澄清可以形成勘误，但不得改变冻结语义；
- 具体实现方案可以演进，但不得降低成熟责任、用户主权、本地权威和真实性边界；
- 新发现若证明根定义存在缺陷，应形成新版本，不得静默修改 V0.2；
- 工程实现、测试通过或用户验收必须另行记录；
- 成熟系统架构不得覆盖或替代本定义；
- V0.2 必须作为历史冻结锚点长期保留；
- 后续成熟判据只能操作化本定义，不能将成熟退化成功能清单或系统数量验收。
