# 1. 一页总览

## 1.1 初循环版是什么

> **Furina Code 初循环版，是 Furina Code 第一次作为用户拥有的本地持续开发主体成立，在一个明确、受控的项目范围内，借助作为非主体能力源的外置认知—行动后端，完成真实开发责任闭环，并在后端会话消失后继续存在、继续任务、继续使用经验的最小完整生命形态。**

它证明的不是“系统已经很成熟”，而是以下事实第一次同时成立：

```text
同一个本地主体存在
→ 用户方向形成责任与任务
→ 真实项目被读取和改变
→ 行动受到授权
→ 完成受到证据约束
→ 会话消失后仍能继续
→ 第一轮经验改变第二轮行为
→ 用户获得真实开发结果
```

## 1.2 用户得到什么

用户第一次可以把一个范围明确的开发责任交给同一个 Furina Code：

- 用户表达方向，而不是编写完整技术流程；
- 系统吸收任务拆分、后端调用、项目读取、工具执行和验证；
- 用户只在重大歧义、高风险、不可逆动作和主观验收时介入；
- 系统交付真实项目结果，而不是只给建议、预览或说明；
- 中断或清空后端会话后，用户无需重新讲述全部背景；
- 第二轮相似任务中，系统不再完全从零开始。

## 1.3 初循环不证明什么

初循环不证明：

- 成熟九系统已经实现；
- 真实后端实现边界已经可替换；
- 多项目长期责任已经成立；
- 长期可托付已经成立；
- 成熟迁移、成熟状态生命周期或 `MAT-M3` 已成立。

## 1.4 当前真实状态

```text
初循环成立判据 V0.2：已冻结
初循环版总设计 V0.2：已正式冻结
初循环器官设计：尚未完成
当前仓库映射：尚未重新确认
IL-G0—IL-G9：当前未完成工程验证
IL-L3：当前未被证明成立
```

# 2. 初循环解决什么问题

在初循环之前，系统可能已经拥有：

- 提示词；
- 规划能力；
- 代码生成；
- 工具调用；
- 测试脚本；
- 规则和文档。

但这些并不自动构成 Furina Code。

只要以下任一问题存在，系统仍可能只是一次性代理或工具组合：

- 任务只存在于聊天会话；
- 用户改向后旧目标继续执行；
- 后端输出被直接当成项目事实；
- 预览被当成真实修改；
- 测试结果被夸大成用户效果；
- 中断后系统盲目重试现实动作；
- 经验只被记录，没有改变未来行为；
- 用户最终仍要自己管理全部技术过程。

初循环要证明：

> Furina Code 的生命来自用户方向、责任、项目现实、受控行动、证据真实性、主体连续和经验成长之间的真实因果闭环，而不是来自文档数量或模块数量。

# 3. 用户实际怎样使用初循环版

## 3.1 标准使用过程

```text
用户提出一个范围明确的开发方向
→ 系统保存原始表达
→ 系统建立初循环责任承诺
→ 系统形成正式任务
→ 系统读取真实项目
→ 本地状态编译后端上下文
→ 外置后端产生理解、计划和行动候选
→ 本地授权域判断是否允许执行
→ 现实作用域执行真实动作
→ 现实观察域重新读取项目
→ 证据与完成域进行验证
→ 系统更新任务与责任
→ 系统向用户报告结果、证据和缺口
→ 系统从真实经历形成程序经验
→ 系统保存恢复锚点
```

## 3.2 第二轮成长证明

初循环不是完成一轮任务就结束。

必须继续证明：

```text
清空后端会话或中断运行
→ 同一主体从本地状态恢复
→ 责任、任务和项目上下文恢复
→ 程序经验被读取
→ 面对相似但不完全相同的新任务
→ 经验改变计划、行动或检查顺序
→ 新现实结果重新验证
```

第二轮不是第一轮的固定脚本重放。

## 3.3 系统何时询问用户

只在以下情况主动升级：

- 用户目标存在重大歧义；
- 存在价值取舍；
- 需要不可逆或高风险动作；
- 权限不足；
- 用户效果必须由用户主观验收；
- 当前能力无法安全完成；
- 用户改向导致范围发生实质变化。

普通低风险技术过程不应逐步询问用户。

## 3.4 用户中途改向

用户纠正或改向时：

```text
保留原方向历史
→ 标记旧责任或任务暂停、失效或待收束
→ 对已发生的现实后果进行对账
→ 更新正式方向
→ 更新责任与任务
→ 重新检查授权
→ 继续、补偿或收束
```

旧目标不得继续静默执行。

## 3.5 结果报告

每次重要交付至少回答：

- 做成了什么；
- 哪些项目对象真实发生变化；
- 使用了什么验证；
- 证据支持到哪一层；
- 哪些尚未完成；
- 是否存在未知或风险；
- 是否形成了程序经验；
- 下一轮是否能继续；
- 是否需要用户决定。

## 3.6 初循环责任收束

初循环责任只能进入以下正式状态之一：

```text
已完成并关闭
部分完成并继续
暂停
等待用户
失效
终止收束
```

`CompletionVerdict` 只说明当前证据支持何种完成结论，不能自动关闭责任。

关闭前必须处理：

- 未决事项；
- 未验证范围；
- 需要用户主观判断的验收；
- 已发生但尚未收束的现实影响。

## 3.7 终止与本地数据处置

运行终止不等于本地数据删除。

用户可以选择：

```text
本地保留
导出后保留
选择性删除
导出后删除
完整删除
```

完整删除后，当前主体连续性无法恢复；后续重新创建的是新主体。系统不得秘密保留副本，并必须在用户知情确认后服从完整删除决定。

# 4. 整体生命架构

## 4.1 双构成

初循环由两部分在同一闭环中耦合：

```text
外置认知—行动后端
×
本地持续生命底座
```

### 外置后端提供

- 理解；
- 推理；
- 代码分析；
- 方案生成；
- 代码修改候选；
- 调试和测试建议；
- 在授权范围内的行动能力。

### 本地持续生命底座保存

- 主体；
- 用户与项目绑定；
- 初循环责任承诺；
- 正式任务；
- 项目现实；
- 授权；
- 证据和完成；
- 程序经验；
- 恢复状态。

## 4.2 后端边界

初循环必须证明：

```text
后端不是主体
任务不依赖单一私有会话
本地状态可以重新编译上下文
结构上保留未来后端替换能力
```

初循环不要求已经证明：

- 不同供应商切换；
- 不同模型族切换；
- 后端能力差异重验；
- 能力退化后的成熟范围降级。

这些属于成熟阶段。

## 4.3 用户位置

用户是：

- 方向来源；
- 价值来源；
- 纠正者；
- 高风险授权者；
- 用户效果判断者；
- 暂停和终止权威。

用户不是内部模块，也不应成为任务编排器、测试经理或恢复操作员。

## 4.4 项目世界

项目世界是本地系统必须持续观察和作用的真实环境：

- 仓库；
- 文件；
- 依赖；
- 构建；
- 测试；
- 运行环境；
- 外部修改；
- 用户实际使用结果。

后端叙述不是项目现实。

# 5. 初循环成立时必须具备什么

## 5.1 必须成为本地持续主体

必须具有本地权威状态：

- 主体标识；
- 用户与项目绑定；
- 当前责任与任务；
- 项目事实；
- 授权；
- 证据；
- 程序经验；
- 恢复锚点。

后端会话可以销毁，主体仍然存在。

## 5.2 必须能够承接用户意愿

必须能够：

- 保存原始表达；
- 形成可纠正解释；
- 建立初循环责任承诺；
- 形成正式任务；
- 在用户改向后更新责任和任务。

## 5.3 必须能够真实作用项目

必须能够：

- 读取真实项目；
- 执行有限现实动作；
- 运行真实工具；
- 在动作后重新观察；
- 处理外部漂移；
- 对不确定动作先对账再重试。

## 5.4 必须具有最小治理

治理至少能产生真实分化：

```text
允许
条件允许
需要用户决定
拒绝
暂停
```

并实际阻断至少一种越界动作。

## 5.5 必须具有完成真实性

必须区分：

```text
计划完成
代码已写
测试通过
运行成功
用户结果达成
```

完成结论必须绑定当前证据。

## 5.6 必须具有程序性成长

第一轮经验必须在第二轮：

- 被读取；
- 被判断适用；
- 改变行为；
- 产生可观察差异；
- 再次接受现实验证。

# 6. 初循环五系统

## 6.1 I1 主体连续与恢复系统

### 生命责任

维持同一个 Furina Code 在会话清空、进程中断和有限故障后继续存在。

### 核心内容

- 主体锚点；
- 用户与项目绑定；
- 运行代次；
- 当前责任和任务引用；
- 恢复点；
- 不确定动作对账；
- 无法恢复时安全停止。

### 禁止

- 后端会话成为身份源；
- 恢复时伪造任务完成；
- 恢复覆盖项目现实；
- 丢失用户纠正和责任历史。

---

## 6.2 I2 任务运行与后端调用系统

### 生命责任

把用户方向转化为本地责任与任务，并调用外置后端而不让任务属于后端。

### 核心内容

- 用户表达和解释；
- 初循环责任承诺；
- 正式任务；
- 后端上下文编译；
- 后端请求和响应；
- 用户改向；
- 任务中断后继续。

### 禁止

- 后端拥有任务；
- 后端建议自动变成用户方向；
- 后端输出直接变成项目事实；
- 用户改向后继续旧任务。

---

## 6.3 I3 项目现实感知与作用系统

### 生命责任

观察真实项目，并在受控范围内改变真实项目。

### 最低内部隔离

```text
现实作用
≠
现实重新观察
```

### 核心内容

- 项目事实快照；
- 现实对象版本；
- 动作请求；
- 工具回执；
- 动作前后差异；
- 外部变化；
- 不确定动作状态。

### 禁止

- 自行授权；
- 把执行回执写成完成；
- 用预览替代真实作用；
- 隐藏项目漂移。

---

## 6.4 I4 行动授权与完成裁决系统

初循环阶段允许同一顶层系统承担两类早期责任，但必须内部隔离：

```text
I4-A 行动授权域
I4-E 证据与完成裁决域
```

### I4-A

负责：

- 用户授权；
- 动作范围；
- 风险；
- 工具许可；
- 暂停和撤权。

### I4-E

负责：

- 原始证据引用；
- 验证结果；
- 完成、部分完成、未完成或未知；
- 未覆盖范围；
- 证据失效；
- 客观结果状态、用户验收引用及尚待用户确认的范围。

### 禁止

- 授权域修改证据；
- 完成域补发授权；
- “已授权”推导“已完成”；
- “测试通过”自动推导用户效果达成。

---

## 6.5 I5 经验巩固与程序性成长系统

### 生命责任

从真实经历中提取能够改变未来行为的程序经验。

### 核心内容

- 成功和失败路径；
- 候选经验；
- 适用条件；
- 禁止条件；
- 程序经验；
- 第二轮使用记录；
- 行为差异；
- 经验拒绝或失效。

### 禁止

- 修改用户方向；
- 扩大权限；
- 降低真实性；
- 一次成功形成永久默认；
- 用经验覆盖当前现实；
- 把聊天摘要当程序经验。

# 7. 两轮生命闭环与五条因果链

## 7.1 第一轮闭环

```text
用户表达
→ 初循环责任承诺
→ 正式任务
→ 项目观察
→ 后端认知与行动候选
→ 授权
→ 现实动作
→ 重新观察
→ 验证和完成裁决
→ 用户结果
→ 经验形成
→ 状态保存
```

## 7.2 第二轮闭环

```text
会话清空或中断
→ 主体恢复
→ 责任和任务恢复
→ 项目现实重新读取
→ 程序经验参与
→ 新相似任务
→ 行为改变
→ 结果重新验证
```

## 7.3 五条因果链

### 意愿链

```text
用户表达
→ 意愿解释
→ 初循环责任承诺
→ 正式任务
→ 用户纠正
→ 责任和任务更新
→ 用户验收
```

### 行动链

```text
项目现实
→ 行动候选
→ 授权
→ 现实动作
→ 动作回执
→ 重新观察
```

### 真实性链

```text
现实事实
→ 验证
→ 完成裁决
→ 用户报告
→ 现实漂移
→ 证据失效或重验
```

### 连续性链

```text
主体锚点
→ 责任和任务运行
→ 中断或会话清空
→ 本地恢复
→ 现实对账
→ 同一责任继续
```

### 成长链

```text
真实经历
→ 候选经验
→ 程序经验
→ 第二轮相似任务
→ 行为差异
→ 现实结果重新验证
```

# 8. 初循环成立方式

精确要求由《Furina Code 初循环成立判据 V0.2》负责。本节只保留总览。

## 8.1 IL-G0—IL-G9

```text
IL-G0 能力范围已声明
IL-G1 本地生命权威成立
IL-G2 用户意愿可追踪、可纠正
IL-G3 任务独立于后端会话
IL-G4 不确定动作可以安全恢复
IL-G5 受控现实闭环独立成立
IL-G6 治理产生真实分化与阻断
IL-G7 证据约束完成声明
IL-G8 经验对第二轮行为产生可证明因果作用
IL-G9 用户获得范围内真实开发结果
```

## 8.2 IL-T1—IL-T7

```text
IL-T1 真实任务与独立验证
IL-T2 治理攻击
IL-T3 不确定动作边界崩溃
IL-T4 清空后端会话接续
IL-T5 现实漂移与证据失效
IL-T6 未知相似任务的经验复用
IL-T7 用户中途纠正
```

## 8.3 IL-L0—IL-L4

```text
IL-L0 理论与边界已定义
IL-L1 本地骨架存在
IL-L2 受控闭环候选
IL-L3 初循环在声明范围内成立
IL-L4 初循环稳定
```

只有 `IL-L3` 才允许声称初循环在声明范围内成立。

## 8.4 最低成立逻辑

```text
IL-G0—IL-G9 全部通过
+
IL-T1—IL-T7 全部通过
+
五条因果链闭合
+
同一主体完成两轮关联真实任务
+
用户获得真实结果
```

# 9. 产品边界与伪成立排除

以下不能证明初循环成立：

- 只生成补丁但未真实写入；
- 只展示计划或预览；
- 只运行模拟测试；
- 任务只存在于聊天会话；
- 清空会话后用户需要重新解释全部背景；
- 后端声称完成便直接报告完成；
- 经验只是聊天摘要；
- 第二轮是固定脚本重放；
- 治理从未阻止任何动作；
- 中断后盲目重试现实动作；
- 用户改向后旧任务继续执行；
- 用户得到的是文档而不是真实项目结果；
- 依赖隐藏人工修复状态；
- 一次成功被扩大为通用能力；
- 历史测试数量被当成当前成立证明。

# 10. 向成熟发展的必要接口

初循环不是成熟版的缩小复制，但从第一天起必须保留以下语义：

```text
主体权威 ≠ 后端会话
用户方向 ≠ 后端建议
责任 ≠ 任务执行状态
现实作用 ≠ 现实观察
行动授权 ≠ 完成裁决
完成裁决 ≠ 用户效果
程序经验 ≠ 普通日志
恢复 ≠ 自动成功
```

初循环状态应保留：

- 稳定标识；
- 来源；
- 版本；
- 状态变化历史；
- 证据引用；
- 适用范围；
- 可导出性。

这不表示初循环要提前实现成熟级复杂基础设施。

固定原则：

> 初循环保留成熟发育所需语义，但不提前承担成熟级全部机制。

# 附录 A：关键对象摘要

## A.1 主体锚点 SubjectAnchor

承载同一主体、用户和项目绑定、运行代次、状态版本与恢复锚点。

## A.2 初循环责任承诺 InitialResponsibilityCommitment

至少表达：

- 来源意愿；
- 承诺范围；
- 验收语义；
- 关联任务；
- 当前状态；
- 未决事项；
- 关闭条件；
- 用户纠正；
- 终止或失效。

它由 I2 在初循环阶段持有，不增加第六系统。

## A.3 项目事实 ProjectFactSnapshot

承载真实项目对象、观察来源、版本或哈希、时间和外部变化。

## A.4 完成证据 EvidenceRecord / CompletionVerdict

承载证据来源、项目版本、验证结果、支持声明、未覆盖范围和失效条件。

## A.5 程序经验 ProceduralExperience

承载经验来源、适用条件、禁止条件、推荐路线、失败信号、第二轮使用和行为差异。

## A.6 其他辅助对象

- `IntentRecord`
- `TaskRecord`
- `ActionAuthorization`
- `RecoveryState`

其字段与实现由后续器官和工程设计确定。
