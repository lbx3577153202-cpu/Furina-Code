## 2. P2：五系统与十九个责任器官

### I1 持续自我与恢复

- I1-A 主体—用户—项目绑定；
- I1-B 连续性投影与生命状态账本；
- I1-C 检查点与恢复裁决。

它维持同一主体和跨会话连续性，但不拥有用户方向、项目事实或完成裁决。

### I2 任务与后端中介

- I2-A 意图—任务档案；
- I2-B 后端能力登记与适配边界；
- I2-C 上下文包和披露过滤；
- I2-D 候选收件与任务运行编排。

它保持用户方向和任务不随后端会话消失。**这里的 Adapter 只表示供应商/协议隔离，不决定产品入口必须采用“本地控制器启动后端”的形态。**

### I3 项目现实与受控行动

- I3-A 多源项目观察与快照；
- I3-B 行动计划绑定与预检；
- I3-C 副作用执行与动作收据；
- I3-D 行动后再观察与现实对账。

它能改变项目，但不能扩大目标、授予自己权限或验证自身成功。

### I4 治理、证据与完成

- I4-A 策略与授权判断；
- I4-B 执行强制、票据和撤权 Gate；
- I4-C 证据谱系与关联；
- I4-D 验证计划与证据评价；
- I4-E 完成裁决。

判断、执行、证据、验证和完成必须分离，避免一次模型输出同时获得全部权力。

### I5 经验生长

- I5-A 经验候选提炼；
- I5-B 适用性检索与试用；
- I5-C 晋升、降级与废止。

经验只从已裁决经历中生长，并且只影响候选。

## 3. 七条工程不变量

1. 用户主权不外包；
2. 项目现实不由语言决定；
3. 正式对象单一权威；
4. 策略判断与行动强制分离；
5. 恢复不能盲目重放副作用；
6. 日志不自动等于证据；
7. 经验不得静默改变权威状态。

## 4. P3：五组织

### O1 生命谱系与对象契约

提供统一身份、对象引用、版本和正式事件骨架，但不是全局写入中台。

### O2 任务运行与恢复

维持任务档案、运行阶段、检查点和跨组织恢复协作。

### O3 外部世界与受控改变

包含两个隔舱：

- O3-C：后端认知候选、上下文和外部能力边界；
- O3-P：项目观察、预检、执行、收据和对账。

两者共享外部边界治理，但认知候选通道与副作用通道必须权限隔离。

### O4 治理、证据与完成

负责授权、强制 Gate、证据谱系、验证和完成裁决。

### O5 经验治理与回流

只接收经过 O4 裁决且允许学习的经历，向新任务提供条件化建议。

P3 明确拒绝：全局状态中台、独立事件总线、独立恢复中台、独立证据仓库、把用户变成内部组织、把数据库/Git/MCP/模型当生命组织。

## 5. P4：冻结什么

P4 冻结：

- 五类权威根和优先关系；
- 正式对象唯一 OWNER 和写入入口；
- 最低字段语义和不可变修订；
- 任务、动作、票据、恢复、验证/完成和经验状态机；
- 十一项跨组织协议；
- 幂等、并发、版本、未知副作用、错误和证据规则；
- IL-G0—G9 可执行断言。

P4 不冻结：编程语言、数据库、工作流框架、模型供应商、Git 库、具体哈希或传输协议。

P4 冻结不表示这些契约已经全部编码或通过真实验证。

## 6. IL-G0—G9 责任链

```text
G0 运行绑定
→ G1 意图成案
→ G2 现实充分
→ G3 授权有效
→ G4 候选受限
→ G5 行动可对账
→ G6 证据充分
→ G7 完成诚实
→ G8 中断可恢复
→ G9 经验可控
```

当前 E4 只读闭环覆盖其中受限切片，不能据此宣称全部 Gate 已通过。

## 7. 实现路线

工程采用纵向切片：

```text
可持续记录
→ 可真实观察
→ 可受控改变
→ 可证明成长
```

每轮应从用户方向/任务一直走到现实、证据和结果，即使能力很小。禁止按 I1 全做完、再 I2 全做完的横向孤岛路线。

## 8. 外部能力使用原则

外部数据库、工作流、策略、Git、MCP、模型和检索系统只能作为材料或基础设施：

- 引用前审查来源、许可证、维护、安全和适配；
- 接入后通过明确端口封装；
- 不授予用户主权、正式对象写权、授权权或完成权；
- 原始框架的物种假设与 Furina Code 冲突时必须替换其根逻辑。

## 9. 当前后端入口解释

P2/P3 的“后端中介/适配器”是历史工程术语，不能定义 Furina Code 的产品关系。当前根关系是：后端本来就是 Furina Code 的现实入口、认知载体和行动载体；本地持续生命系统使该入口中的会话持续成为同一主体。

- 入口/载体描述 Furina Code 如何真实存在、与用户交互、认知和行动；
- 任何中介或传输结构只可服务于本地状态在该入口中的持续、披露约束和候选登记，不能把后端降格为可被独立控制器调用的 provider；
- 错误发生在把中介层误做成独立完整主体，再让它 headless 启动 MiMo；
- 正确方向是在后端承载的现实运行中，使本地持续生命系统成为同一 Furina Code 的内在连续结构，而不是另一层产品入口。

这一解释由 `PDR-001` 固定。
