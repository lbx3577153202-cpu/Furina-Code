# Furina Code 开发方式

## 1. 总定义

Furina Code 的开发方式是：

```text
造物台
+
判断/审判
+
执行
+
验证
+
证据
+
安全/Gate
+
记忆成长
+
用户成效汇报
```

它不是固定流水线，而是一套让定义、工程现实和用户结果在同一闭环中相互约束的方法。

## 2. 造物台：从问题生成生命结构

造物台负责：

```text
现实问题
→ 必须维持的生命状态
→ 定义
→ 基因/约束
→ 器官责任
→ 组织协作
→ 契约
→ 工程切片
```

大定义决定物种，小定义决定器官和行为。造物台不能取代判断、执行和证据；它只负责“应该造什么以及为什么”。

## 3. 判断/审判机制

### 构建前判断

- 用户真正要的效果是什么；
- 这是不是当前主线问题；
- 已有能力能否直接满足；
- 是否把未来能力误当当前需求；
- 是否存在更简单直接的路径；
- 新开发的收益是否超过复杂度和维护成本。

### 构建中判断

- 实际变更是否仍在任务范围；
- 设计是否与仓库现实冲突；
- 是否出现权限、证据、依赖或副作用风险；
- 是否应暂停、降级、回滚或请求用户决定。

### 落地后判断

- 代码存在是否真的产生用户效果；
- 测试覆盖什么，没覆盖什么；
- 真实环境是否验证；
- 是否应合并、继续、暂停、归档或删除；
- 对外能声称到哪个能力层级。

本次 MiMo headless 路线的教训是：构建前没有用“用户当前是否直接可用 MiMo Code”审判新路线的必要性，导致造了技术上可运行但当前收益极低的旁路。

## 4. 执行机制

执行必须：

- 以一张有界任务卡或等价正式任务为入口；
- 在隔离分支和明确路径中实施；
- 对副作用使用预检、票据、收据和对账；
- 保留失败和未知状态；
- 不因为“可以重试”就假设上一次没有发生。

## 5. 验证机制

验证从用户成功条件和风险反向生成，不只运行已有测试。

至少区分：

- 合同/类型有效；
- 单元行为；
- 集成路径；
- 真实环境；
- 对抗/故障；
- 用户效果。

验证结果必须包含覆盖、失败、未知、跳过和失效条件。

## 6. 证据机制

证据分四层：

```text
原始材料
→ 来源与关联
→ 验证结论
→ 完成裁决
```

命令输出、diff、测试日志和后端响应只是原始材料。只有和任务、授权、对象、时间、版本和验证要求建立谱系后，才能支持正式结论。

## 7. 安全与 Gate

Gate 的职责是强制边界，不是替用户创造授权。

必须保持：

- 判断与强制分离；
- 票据绑定任务版本、对象、动作、现实和时效；
- 当前现实漂移后重新复核；
- 未知、过期、撤权和证据不足时 fail-closed；
- Gate 通过产生结构化依据，而不是无来源的 `true`。

## 8. 记忆成长

经验来自成功、失败、部分完成和人工介入，必须保存：

- 来源任务和证据；
- 适用与禁止条件；
- 推荐路线和替代路线；
- 失败信号；
- 用户纠正；
- 第二轮是否采用；
- 采用后真实行为差异；
- 晋升、降级和废止理由。

“以后可能有用”不足以保留为当前功能，但可以成为带恢复条件的归档路线。

## 9. 用户成效汇报

对用户默认只汇报：

```text
做成了什么
用户实际获得什么
证据是否足够
还有什么没做成
风险和例外
下一步为什么值得做
```

内部命令、分支、调试和复杂流程由系统吸收；只有用户需要决策或证据不足时才展开。

## 10. 工程迭代单位

每一轮优先选择最小真实纵向切片：

```text
用户方向
→ 正式任务
→ 当前现实
→ 后端候选
→ 判断/授权
→ 行动或只读处理
→ 再观察
→ 验证
→ 完成裁决
→ 经验/汇报
```

切片可以缺少后续成熟机制，但不能用 mock 或文档冒充已经穿过的现实环节。

当切片涉及后端时，必须从 Furina Code 正在使用的现实入口出发，不能默认添加“本地控制器调用后端”的旁路。`BackendPort`、文件交换或其他传输技术只能是入口内的局部工程材料，不能反向定义 Furina Code 的物种关系。

## 11. 每轮完成条件

- 变更进入明确 commit / PR；
- 实际运行的测试和检查被记录；
- 当前 HEAD 和工作树可核验；
- 能力现实账本被更新；
- 未完成、未验证和范围外项明确；
- 若路线改变，PDR 和 Archive Index 同步更新；
- 用户得到低打扰、可理解的结果报告。
